# Tickets: 评估表单维度化重构

将绩效表单从“评估维度 + 评估项”重构为“计分/非计分维度 + 表单字段”，统一员工自评、360°评估和上级评估。来源：[PRD](./PRD.md)。

状态：`ready-for-agent`

按 **frontier** 实施：仅开始所有阻塞项均已完成的 ticket；其中“打通 360°完整链路”和“打通上级评估完整链路”可并行。

## 建立新版表单模板编辑与预览

**What to build:** 管理员可以用“计分维度 / 非计分维度 + 表单字段”创建、编辑、校验、预览并发布员工自评、360°评估和上级评估表单；计分方式、占比和核心维度直接属于维度，表单字段不再包含评级或分数。

**Blocked by:** None — can start immediately.

- [x] 计分维度支持评级或 0～100 分、占比和核心维度；非计分维度不展示也不保存这些配置。
- [x] 同一子表单可混用评级计分维度和分数计分维度。
- [x] 每个人工评估子表单发布时至少有一个计分维度、占比合计 100%、单个占比大于 0 且不超过 100（最多两位小数），并且恰好一个核心维度。
- [x] 表单字段仅支持短文本、长文本、Markdown、单选、多选、附件和链接；计分维度的长文本或 Markdown 字段可按维度等级设置条件必填。
- [x] 维度和字段使用跨版本不变的业务标识；改名、改类型、排序和移动不改变标识，删除后的标识不复用。
- [x] 草稿编辑允许暂时不完整，发布时返回清晰的中文校验错误。
- [x] 默认模板内容保持不变；员工自评默认一个占比 100% 的评级计分核心维度，默认评语字段仅在 S/C 等级条件必填。
- [x] 管理端编辑、预览和错误文案中不再出现“评估项”，并在组件测试页覆盖新增或变更的组件状态。

## 生成新版周期快照与统一计算规则

**What to build:** 管理员从新版模板创建周期时得到可审计的新版快照，并能预览混合计分维度的统一换算与约束结果；人工评估不再配置阶段计算模式、双套约束档位或全局评语必填。

**Blocked by:** 建立新版表单模板编辑与预览。

- [ ] 周期配置、接口和页面不再暴露人工阶段计算模式、评级/分数约束档位和全局评语必填。
- [ ] 周期快照完整复制维度、字段、稳定业务标识、条件必填和评分配置，发布后的模板变化不影响既有周期。
- [ ] 评级和分数输入都换算为 calculationScore，再按评级区间派生维度等级。
- [ ] 统一约束为：核心维度 C 强制 C、核心维度 B 最高 B、任一维度 C 最高 B；红线规则继续独立强制 C。
- [ ] 360°关系聚合顺序和既有业务语义保持不变。

## 打通员工自评完整链路

**What to build:** 员工可以按新版表单完成自评草稿、正式提交和结果查看，系统以维度回答与字段回答保存输入并计算自评阶段结果。

**Blocked by:** 生成新版周期快照与统一计算规则。

- [ ] 填写页在维度上直接展示评级或分数输入，其后展示该维度的表单字段。
- [ ] 草稿允许计分输入或条件必填字段暂未完成，正式提交时完整校验。
- [ ] 每次提交按“提交记录 → 维度回答 → 字段回答”保存，且同一提交中维度和字段回答唯一。
- [ ] 计分维度输入正式提交时一律必填，条件字段根据该维度最终等级校验。
- [ ] 自评阶段结果使用统一加权计算和核心维度约束，并可在员工结果页正确展示。

## 打通 360°完整链路

**What to build:** 360°评估人可以使用新版表单保存草稿和提交评估，系统按原有关系聚合顺序处理新版维度回答与字段回答，并产出统一的阶段结果。

**Blocked by:** 打通员工自评完整链路。

- [ ] 360°填写页正确呈现计分/非计分维度及字段，不再呈现评估项。
- [ ] 草稿、正式提交、必填校验和条件必填遵循新版规则。
- [ ] 多位评估人的维度得分按既有关系聚合顺序汇总，混合评级与分数维度可正确计算。
- [ ] 360°结果与详情使用新版回答结构且保持现有权限边界。

## 打通上级评估完整链路

**What to build:** 直属 Leader 可以使用新版表单保存和提交上级评估，系统按统一加权、等级派生和核心维度约束产出阶段结果。

**Blocked by:** 打通员工自评完整链路。

- [ ] 上级评估填写页正确呈现计分/非计分维度及字段，不再呈现评估项。
- [ ] 草稿、正式提交、必填校验和条件必填遵循新版规则。
- [ ] 混合评级与分数维度统一换算、加权并派生等级。
- [ ] 核心维度和任一 C 维度约束正确限制上级评估结果，红线规则保持独立。

## 迁移结果消费者与结构变更链路

**What to build:** 校准、员工结果、AI 输入、影响预览、周期表单变更和报表等消费者全部读取新版维度/字段回答；管理员在周期内变更表单时可依靠稳定业务标识安全预填兼容数据。

**Blocked by:** 打通 360°完整链路；打通上级评估完整链路。

- [ ] 校准、员工结果、AI 输入和报表不再依赖旧评估项结果。
- [ ] 影响预览能准确说明维度评分配置、字段或条件必填变化的影响。
- [ ] 周期内表单变更使用稳定业务标识匹配旧、新结构，兼容值可预填，不兼容值明确失效且不会错配。
- [ ] 改名、排序、跨维度移动和类型变更不会生成新的业务标识。
- [ ] 已发布结果、导出和审计展示采用统一的新领域术语。

## 删除旧模型、重置数据并完成集成验收

**What to build:** 系统彻底移除旧评估项、旧结果和人工阶段模式等兼容结构，精确清理开发期绩效业务数据并用新版种子重建基线，最后完成端到端验收。

**Blocked by:** 迁移结果消费者与结构变更链路。

- [ ] 删除旧评估项、旧评估项结果、人工阶段模式、双套约束档位和全局评语必填相关模型、接口、页面和文案。
- [ ] 既有晋升表单内容及历史保持只读，并排除在新版绩效模板发布、快照和提交之外；本 ticket 不实现新的晋升流程。
- [ ] 仅删除所有 `perf_*` 绩效业务数据及其关联审计日志、导出任务，保留 `lark_*`、`role_grants` 和 `system_configs`。
- [ ] 新版基线种子可以从空绩效业务数据重建默认模板和必要配置。
- [ ] 后端构建与完整测试、前端 lint/build、组件测试页和员工自评/360°/上级评估关键流程全部通过。
