# 评估表单维度化重构

Status: ready-for-agent

## Problem Statement

当前员工自评、360°评估和上级评估的表单设计规则不统一。系统把评估维度作为分组和权重单元，却又在每个计分维度下强制创建唯一的评级或分数“评估项”；非计分的文本、选项、附件和链接也使用同一个评估项概念。这让管理员必须理解“维度 → 唯一计分项 → 其他评估项”的三层结构，而填写人实际感知到的评分对象仍然是维度。

现有阶段模式进一步放大了重复：360°与上级评估在配置模板中先选择“加权评级”或“加权评分”，每个维度内又保存对应评级或分数评估项。同一表单不能根据评价内容混用评级与分数，员工自评还保留独立直接定级例外。评级约束和分数阈值分别维护，可能与统一评级区间发生漂移。

作答存储同样沿用旧概念。评级、分数、文本和附件全部保存为评估项结果；所谓稳定 key 又由子表单、填写对象和排序位置生成，管理员换序后可能改变对象身份，影响结构差异判断和旧值预填。全局评级定义中的“评语必填”也没有稳定指向某个具体文本字段。

从用户角度看，这些问题导致：

- Admin 难以快速理解一个维度到底在哪里设置计分方式、占比和核心规则。
- 同一业务内容在编辑器、预览、填写页、发布校验和计算服务中使用不同表达。
- 员工、360°评审员和 Leader 看到不必要的“评估项”层级。
- 混合评级与分数需要额外拆模板或改变整个阶段模式。
- 条件必填、结构变更、答案预填和计算数据血缘缺少稳定边界。
- 晋升申请表单仍绑定在绩效评估模板中，尽管晋升已经被确认为员工收到资格邀请后自行发起的独立流程。

## Solution

将三类绩效评估表单统一重构为“评估维度 + 表单字段”两层模型。

评估维度分为计分维度和非计分维度。计分维度直接拥有计分方式、占比和核心标记，计分方式可以是评级或 0～100 分；同一子表单允许不同维度混用两种方式。非计分维度没有计分方式、占比和核心标记，可以包含表单字段，也可以只展示说明。

表单字段只承载非计分内容，包括单行文本、多行文本、Markdown、单选、多选、附件和链接。评级与分数不再作为字段。字段支持选填、始终必填，以及计分维度下多行文本或 Markdown 按指定维度等级条件必填。

员工自评、360°评估和上级评估统一使用维度加权计算。每个子表单至少包含一个计分维度，单个计分维度占比必须大于 0%、不超过 100% 且最多两位小数，全部计分维度占比严格合计 100%，并且有且只有一个核心维度。默认员工自评使用一个占比 100% 的评级核心维度，从而保持当前自评结果语义。

所有评级和分数先统一形成 0～100 计算分，再按评级区间派生维度等级。核心维度和任一维度约束只使用派生等级，不再维护评级模式与分数模式两套约束。三类人工阶段模式从配置模板和周期配置中删除。

作答数据改为“评估提交 → 维度作答 → 字段作答”。原始评级、原始分数和计算分归属维度作答；文本、选项、附件和链接归属字段作答。维度和字段使用跨版本不可变业务 key，身份不再依赖名称、类型、排序或数据库行 ID。

管理员编辑器、预览和三类填写页统一按维度展示计分控件与补充字段，并全面移除“评估项”称呼。默认 D/M 表单的维度名称、占比和评价文案保持不变，只迁移到新结构。

项目仍处开发阶段，因此本次切换直接重置旧绩效业务数据并用新版基线脚本重建，不实现旧评估项模型兼容。飞书同步数据、权限授权和系统配置必须保留。

晋升流程不在本次实现。旧晋升表单内容与历史数据只读保留，退出新的绩效模板发布、周期快照和三类评估提交链，等待后续迁移为独立晋升申请表单模板。

## User Stories

1. As an Admin, I want 评估表单只使用评估维度和表单字段两层结构, so that 我不需要理解唯一计分评估项这一层。
2. As an Admin, I want 在维度上直接选择计分或非计分, so that 维度的业务作用一目了然。
3. As an Admin, I want 在计分维度上直接选择评级或 0～100 分, so that 计分控件与评价对象保持一致。
4. As an Admin, I want 同一子表单中的不同维度可以混用评级和分数, so that 每类评价内容都能使用合适输入方式。
5. As an Admin, I want 在计分维度上直接填写占比, so that 不需要在维度和评估项之间寻找权重来源。
6. As an Admin, I want 计分维度占比最多保留两位小数, so that 配置精度统一且计算可复现。
7. As an Admin, I want 单个计分维度占比必须大于 0%, so that 不会出现必须评分却完全不参与计算的维度。
8. As an Admin, I want 同一子表单的计分占比严格合计 100%, so that 每份正式表单都使用完整明确的计算口径。
9. As an Admin, I want 每个子表单必须且只能有一个核心计分维度, so that 核心约束始终有唯一目标。
10. As an Admin, I want 非计分维度不显示计分方式、占比和核心选项, so that 无效配置无法被创建。
11. As an Admin, I want 非计分维度可以没有字段, so that 我可以创建只展示说明或评价指引的内容区。
12. As an Admin, I want 计分维度可以没有补充字段, so that 只需要评分的简单维度保持简洁。
13. As an Admin, I want 在任意维度中添加多个非计分表单字段, so that 评语、选项、附件和链接可以围绕同一评价主题组织。
14. As an Admin, I want 字段类型列表不再包含评级和分数, so that 计分能力不会重新退回字段层。
15. As an Admin, I want 表单字段可以配置为选填或始终必填, so that 常规完整性规则可以按字段表达。
16. As an Admin, I want 计分维度中的多行文本或 Markdown 可以按维度等级条件必填, so that S/C 等特殊等级可以要求具体说明。
17. As an Admin, I want 非计分维度不能配置等级条件必填, so that 没有维度等级时不会产生悬空规则。
18. As an Admin, I want 全局评级配置不再包含评语必填开关, so that 必填规则明确指向具体字段。
19. As an Admin, I want 草稿允许占比暂时不满 100% 或缺少核心维度, so that 我可以分步骤设计模板。
20. As an Admin, I want 发布模板时看到明确的占比、核心、计分方式和字段规则错误, so that 我能快速定位并修复问题。
21. As an Admin, I want 已发布模板继续保持不可变, so that 已被周期使用的表单不会被原地改写。
22. As an Admin, I want 从已发布版本创建草稿时继承维度和字段身份, so that 跨版本结构比较可靠。
23. As an Admin, I want 重命名或换序不改变维度和字段身份, so that 纯展示调整不会丢失答案匹配关系。
24. As an Admin, I want 改变字段或维度类型时保留身份但标记结构不兼容, so that 系统能明确区分对象身份与值兼容性。
25. As an Admin, I want 删除后的稳定 key 永不复用, so that 历史答案不会错误附着到后来新增的对象。
26. As an Admin, I want 编辑器标题行直接展示维度名称、类型、计分方式、占比和核心标记, so that 常用配置无需展开多层面板。
27. As an Admin, I want 展开维度后只编辑说明和表单字段, so that 配置层次保持清晰。
28. As an Admin, I want 切换维度类型时收到不兼容配置清理提示, so that 我不会误以为旧计分设置仍然生效。
29. As an Admin, I want 模板预览与真实填写页使用相同的维度结构, so that 发布前看到的内容可信。
30. As an Admin, I want 管理页面和错误文案全面使用“表单字段”而不是“评估项”, so that 领域语言统一。
31. As an employee, I want 在自评维度上直接选择评级, so that 我不需要理解评级评估项。
32. As an employee, I want 默认自评评级维度占比 100% 且为核心, so that 新模型保持现有自评等级含义。
33. As an employee, I want 在自评表中继续填写总结、规划、支持需求、附件和链接, so that 结构重构不会减少原有内容。
34. As an employee, I want 草稿可以缺少计分输入或必填字段, so that 我可以逐步完成自评。
35. As an employee, I want 正式提交时所有计分维度必须完成, so that 自评结果使用完整口径。
36. As an employee, I want 条件必填字段在选择相应等级后立即提示, so that 我能在提交前补充所需说明。
37. As a peer reviewer, I want 360°表单按维度直接展示评级或分数控件, so that 填写体验与评价主题一致。
38. As a peer reviewer, I want 默认 D/M 360°维度、占比和说明保持不变, so that 模型重构不会改变评价制度。
39. As a peer reviewer, I want 每个维度的补充评价字段紧跟该维度计分控件, so that 评分依据和评语不会分散。
40. As a peer reviewer, I want 正式提交时所有计分维度都必须完成, so that 关系平均不会混入不完整答卷。
41. As an assessment Leader, I want 上级评估维度直接使用 0～100 分或评级, so that 不再受整个阶段统一计分方式限制。
42. As an assessment Leader, I want 默认 D/M 上级评估维度、占比和说明保持不变, so that 制度内容不会被技术重构改变。
43. As an assessment Leader, I want 分数维度先派生维度等级再执行条件必填和核心约束, so that 分数与评级使用统一规则。
44. As an evaluator, I want 计分输入在正式提交时固定必填且没有选填开关, so that 缺失值不会被当成零分或重新归一化。
45. As an evaluator, I want 非计分字段按照各自必填规则校验, so that 可选反馈不会无意阻止提交。
46. As an evaluator, I want 非计分说明维度不生成空作答, so that 答卷只保存真实输入。
47. As an HR, I want 三类人工评估固定使用统一维度加权计算, so that 不需要再解释阶段模式。
48. As an HR, I want 配置模板中移除 SELF、PEER、MANAGER 阶段模式, so that 规则不会与维度计分方式重复。
49. As an HR, I want AI 建议直接产生等级但不暴露阶段模式配置, so that 管理配置只包含真正可调整的规则。
50. As an HR, I want 评级维度按映射分参与加权而分数维度使用原始分数, so that 混合计分仍然落在统一 0～100 标尺上。
51. As an HR, I want 每个计分维度都从计算分派生维度等级, so that 核心和任一维度约束拥有统一判断口径。
52. As an HR, I want 核心维度为 C 时阶段等级强制为 C, so that 核心短板不能被平均分掩盖。
53. As an HR, I want 核心维度为 B 时阶段等级最高为 B, so that 核心表现限制整体上限。
54. As an HR, I want 任一计分维度为 C 时阶段等级最高为 B, so that 明显短板被稳定反映。
55. As an HR, I want 评级区间成为等级边界的唯一来源, so that 不再维护可能漂移的分数阈值。
56. As an HR, I want 360°继续先按评审关系聚合再应用维度占比, so that 本次表单重构不改变已确认的关系加权口径。
57. As an auditor, I want 每条维度作答保留原始评级或原始分数和计算分, so that 阶段结果可以解释和复算。
58. As an auditor, I want 每条字段作答关联明确的维度和稳定字段 key, so that 非计分内容来源可追溯。
59. As an auditor, I want 同一提交内同一维度最多一条作答, so that 不会出现相互竞争的计分结果。
60. As an auditor, I want 同一维度作答内同一字段最多一条作答, so that 字段值没有重复歧义。
61. As an auditor, I want 表单快照保存维度计分方式、占比、核心、字段和稳定 key, so that 历史计算不依赖当前模板。
62. As an implementer, I want 模板定义、周期快照、提交和结果使用同一领域层级, so that 不需要构造虚假的评级或分数字段。
63. As an implementer, I want 计算服务直接读取维度作答, so that 不需要在维度下搜索唯一计分项。
64. As an implementer, I want 动态字段内容保留受控 JSON 载荷但计分值关系化, so that 灵活表单和数据库约束能够兼顾。
65. As an implementer, I want 开发数据库直接使用新模型重建绩效数据, so that 不需要维护旧评估项兼容分支。
66. As an implementer, I want 飞书同步数据在开发重置中保留, so that 组织与员工主数据无需重新构造。
67. As an implementer, I want 角色授权和系统配置在开发重置中保留, so that 重构不会破坏环境访问与基础设置。
68. As an implementer, I want 基线脚本重建默认表单、配置和周期, so that 重置后环境立即可验证。
69. As an employee eligible for promotion, I want 在本期绩效结果发布时同时收到晋升资格通知, so that 我知道自己可以主动申请。
70. As an employee eligible for promotion, I want 只有当前与紧邻上一半年度结果都属于 A/S 时获得邀请, so that 晋升门槛使用连续绩效表现。
71. As an employee, I want 晋升资格通知只提供申请入口而不自动创建流程, so that 是否申请由我本人决定。
72. As an employee, I want 提交独立晋升申请表单后才创建晋升流程, so that 绩效评估和晋升申请保持分离。
73. As an employee, I want 绩效改判导致资格不再满足时收到撤销或终止通知, so that 我不会继续依赖失效资格。
74. As an HR, I want 晋升申请不阻塞绩效周期归档, so that 两个流程可以独立闭环。
75. As an Admin, I want 旧晋升表单内容只读保留并退出新绩效模板链, so that 数据不丢失且错误结构不再扩散。

## Implementation Decisions

- 继续使用现有绩效业务域，不为三类人工评估新建平行业务模块。模板、周期快照、评估提交、计算、结果和前端填写入口共同切换到新模型。
- 绩效评估表单模板只包含员工自评、360°评估和上级评估三个子表单。旧晋升表单不进入新的发布校验、周期快照或绩效提交。
- 评估维度类型固定为计分和非计分。删除旧的常规、文本、晋升三分法。
- 计分维度直接拥有计分方式、占比和核心标记。计分方式固定为评级或 0～100 分；不允许任意脚本或自定义组件。
- 非计分维度不保存计分方式、占比和核心标记。它可以包含零到多个表单字段；零字段时仅作为说明内容区。
- 表单字段类型固定为单行文本、多行文本、Markdown、单选、多选、附件和链接。评级与分数从字段类型目录中删除。
- 三类人工子表单必须至少有一个计分维度。单个计分维度占比必须大于 0%、不超过 100% 且最多两位小数，全部计分维度严格合计 100%。
- 每个三类人工子表单必须且只能有一个核心计分维度。非计分维度不能成为核心。
- 草稿允许结构暂时不完整；模板发布和周期启动检查执行完整业务校验。
- 计分输入不提供必填开关。草稿可缺失，正式提交时所有计分维度必须完成；不允许按零分处理或对剩余维度重新归一化。
- 字段必填规则支持选填和始终必填。计分维度中的多行文本或 Markdown 额外支持按指定维度等级条件必填；其他字段类型和非计分维度不得配置等级条件。
- 评级定义删除全局评语必填属性。评级仍固定为 S/A/B/C，名称、说明、连续分数区间和映射分继续可配置。
- 员工自评、360°评估和上级评估固定使用统一维度加权计算。删除配置模板、周期配置、接口和管理端中的三类人工阶段模式。
- 同一子表单允许评级计分维度与分数计分维度混用。评级按当期映射分转为计算分，分数的计算分等于原始分数。
- 所有计分维度都按当期评级区间从计算分派生维度等级。约束固定为核心 C 强制 C、核心 B 封顶 B、任一 C 封顶 B；红线强制 C 保持独立硬约束。
- 删除评级模式和分数模式两套约束档案以及独立分数阈值。评级区间是维度等级的唯一边界。
- 360°关系内平均、有效关系归一化、关系加权和维度加权顺序保持不变。
- 默认员工自评使用一个评级计分维度，占比 100% 且为核心；现有总结与规划内容迁移为非计分维度和字段。
- 默认 D/M 360°与上级评估的维度名称、占比、核心标记、默认计分方式和评价文案保持不变。默认评价说明字段在 S/C 时条件必填。
- 模板维度与字段各自持久化不可变业务 key。新建时生成，复制版本时继承，重命名、修改说明、改变类型、换序或移动时不改变，删除后不复用。
- 周期表单快照升级结构版本，直接保存维度计分配置和字段列表；快照 key 必须来自模板稳定 key，禁止再按排序位置生成。
- 评估提交继续作为 SELF、PEER、MANAGER 答卷父记录。其下改为维度作答；字段作答从属于维度作答。
- 维度作答保存维度 key、原始评级或原始分数以及计算分。同一提交内维度 key 唯一，原始评级与原始分数互斥且必须匹配维度计分方式。
- 字段作答保存字段 key、字段类型快照和受控结构化值。同一维度作答内字段 key 唯一，值必须匹配快照中的字段类型。
- 非计分且没有已填写字段的维度不创建空维度作答。计分维度正式提交后必须有维度作答。
- 模板保存、发布、周期快照复制、提交 DTO、草稿自动保存、正式提交、结构变更预填、阶段计算、结果投影和报表接口全部改用维度/字段术语与结构。
- 管理编辑器的维度标题行直接展示名称、计分/非计分、评级/分数、占比和核心。展开区只展示维度说明与字段列表。
- 切换维度或字段类型时必须提示并清理不兼容配置；稳定 key 保持不变，值兼容性由结构变更规则判断。
- 填写页先展示维度标题、说明、占比和核心标记，再展示维度计分控件及该维度字段。非计分维度只展示说明与字段。
- 管理页面、预览、填写页、API 文案、错误信息和审计摘要全面使用“评估维度、表单字段、维度作答、字段作答”，不再使用“评估项”。
- 晋升是员工自行申请发起的独立流程。绩效结果版本发布时，以当前结果与紧邻上一半年度有效最终结果判断连续 A/S 资格，并与结果通知一起发送邀请。
- 晋升资格邀请不自动创建流程。员工提交晋升申请表单后才创建；改判后每次结果版本发布都重新校验资格，不满足时撤销入口或终止已发起流程并保留审计。
- 本次不实现新晋升流程或独立晋升模板。旧晋升表单内容和历史数据只读保留，并在管理端作为待迁移内容单独展示。
- 数据库切换不做旧绩效数据迁移或双写。删除全部绩效业务记录及其关联操作日志和导出任务，保留飞书同步数据、角色授权和系统配置。
- 旧评估项定义、旧评估项结果、旧人工阶段模式和旧约束档案从新 Schema 与运行时代码中删除，不保留兼容读取。
- 基线初始化必须同步更新，并幂等重建默认 D/M 表单模板、默认配置模板和开发基线周期。
- Prisma Schema 与数据库表结构文档必须在同一变更中保持字段、约束、索引和删除行为完全一致。

## Testing Decisions

- 测试只断言外部业务行为，不断言私有辅助函数、内部调用次数或组件实现细节。关键结果是模板能否发布、快照是否正确、答卷能否提交、计算结果是否正确以及用户能否看懂并完成交互。
- 首选一个最高层后端测试接缝：通过公开业务服务或 API 完成“创建并发布表单模板 → 绑定配置并生成周期快照 → 提交答卷 → 生成阶段结果”。该接缝应覆盖模型契约在模块之间的真实传递。
- 在主接缝中至少覆盖一个混合计分模板：评级维度与分数维度共同产生统一计算分、综合分、派生等级和约束结果。
- 通过模板发布行为测试计分占比范围、两位小数、合计 100%、唯一核心、非计分字段规则、条件必填合法性及三类子表单完整性。
- 通过提交行为测试草稿可不完整、正式提交计分输入固定必填、字段始终必填、S/C 条件必填、未知或伪造 key 拒绝、字段类型与载荷不匹配拒绝。
- 通过计算行为测试评级映射分、原始分数、混合维度加权、最终两位舍入、统一维度等级、核心 C 强制 C、核心 B 封顶 B、任一 C 封顶 B 和红线强制 C。
- 保留并更新 360°高层计算测试，证明关系内平均、有效关系归一化与关系权重顺序没有因表单重构改变。
- 通过结构变更行为测试稳定 key：重命名和换序保持身份；新增生成新 key；删除不复用；类型变化保留 key 但旧值按兼容性规则处理。
- 通过数据库 Schema 测试验证维度作答与字段作答的外键、唯一约束、互斥计分值、索引和级联/限制删除策略。
- 前端管理员编辑器使用现有组件测试接缝，覆盖计分/非计分切换、计分方式、占比、核心、字段类型、条件必填、错误定位和预览，不测试内部 state helper。
- 三类填写页优先复用统一评估表单组件测试接缝，覆盖评级与分数渲染、字段顺序、条件必填提示、草稿和正式提交载荷。员工自评、360°和上级评估页面只补角色差异测试。
- 复用现有模板发布校验、表单服务、统一提交、360°阶段结果、上级阶段结果、管理员模板编辑器和统一填写组件测试作为 prior art；更新断言到新领域术语和数据结构。
- 组件测试页必须补充新版维度编辑器和填写表单示例，覆盖计分、非计分、混合计分、条件必填、禁用态和校验错误，并进行人工视觉检查。
- 开发数据重置后运行基线初始化，成功标志是默认 D/M 模板、默认配置模板和开发基线周期能够创建并通过发布/启动校验。
- 后端验证至少包含 Prisma Client 生成、Schema 测试、相关单元/服务测试、完整构建和基线初始化。前端验证至少包含相关组件测试、lint 和生产构建。
- 如果主业务接缝无法覆盖前端视觉行为，允许保留一个前端组件接缝作为必要补充；不为每个页面建立重复的端到端测试套件。

## Out of Scope

- 开发新的独立晋升流程、晋升申请状态机、审批角色、晋升结果或独立晋升模板管理。
- 把旧晋升表单正式迁移为新的晋升申请表单模板；本次仅只读保留并退出绩效链路。
- 改变现有 D/M 默认评估维度名称、占比、核心选择或评价文案。
- 改变固定 S/A/B/C 评级符号和顺序。
- 新增评级、分数、文本、Markdown、单选、多选、附件和链接之外的控件类型。
- 支持任意公式、脚本、自定义 HTML 或管理员自定义等级约束类型。
- 改变 360°评审员关系分类、基础关系占比或关系聚合顺序。
- 改变校准、结果确认、申诉、周期退回和归档的既有权威边界。
- 为生产环境提供旧评估项数据的无损迁移、双写、灰度读取或回滚兼容层。
- 删除或重建飞书同步数据、角色授权或系统配置。
- 将季度或年度周期纳入晋升“连续两次”资格计算；当前只定义半年度周期。

## Further Notes

- 本规格使用领域术语：评估维度、计分维度、非计分维度、表单字段、维度作答、字段作答、计算分、维度等级、阶段等级。
- 关键领域决策已经记录在 ADR-0061 至 ADR-0071。ADR-0064 与 ADR-0065 是晋升边界讨论中的已取代决策，实施必须以 ADR-0066 为准。
- 旧 ADR 中的“评估项”“加权评级/加权评分阶段模式”“员工自评直接定级”和“晋升嵌入绩效表单”等内容已被新 ADR 部分或全部取代，不能作为实现依据。
- 开发数据重置是已确认的破坏性操作。执行时必须使用精确表集合，禁止清空整个数据库或整个 PostgreSQL schema；完成后需报告删除范围和恢复条件。
- 主测试接缝按现有代码结构选择公开业务服务/API，而不是新增测试专用入口。前端组件测试是视觉与交互的补充接缝。
- 本规格本身不授权在生成规格阶段执行数据清理或代码重构；后续实现任务开始后才执行已确认的重置与切换。
