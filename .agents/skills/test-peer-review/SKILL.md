# 测试 360° 评估

用 Cursor Browser 打开 360° 评估填写页，按以下顺序填写三个维度（评级 + 评语）并保存草稿。默认只保存草稿，不提交；仅当用户明确要求时才点「提交评估」。

多人造数、线上批量、随机 A/B → 用 `batch-peer-review-api`（不走浏览器）。

## URL

优先使用用户给出的完整 URL。未给出时使用：

```
http://localhost:3001/review-tasks/fill?participant_id=<id>&type=REVIEW&assignment_id=<id>
```

必需参数：`type=REVIEW`、`assignment_id`。`participant_id` 可选但常见。

成功标志：页头为「360°评估」，右侧出现三个评级区与「保存草稿」「提交评估」。

## 操作流程

```
- [ ] 1. browser_tabs list；已有标签则先 browser_lock
- [ ] 2. browser_navigate 打开目标 URL（无标签时 navigate → lock）
- [ ] 3. browser_snapshot + screenshot 确认表单已加载（非「正在加载」）
- [ ] 4. 工作贡献与责任担当：选 A · 优秀 → 填写评语
- [ ] 5. 协作沟通与价值观：选 A · 优秀 → 填写评语
- [ ] 6. 学习成长与潜力：选 A · 优秀 → 填写评语
- [ ] 7. 点击「保存草稿」
- [ ] 8. 等待按钮从 disabled 恢复可用
- [ ] 9. 刷新页面，确认三项评级与三段评语回填
- [ ] 10. screenshot 后 browser_lock unlock，向用户汇报
```

并行填写时注意：先填完当前维度评语再点下一维度评级，避免 controlled textarea 值被覆盖。

## 字段填写规范

三个维度结构相同：必填 `RATING`（C/B/A/S）+ 可选长文本评语。以同部门同事视角写可观察行为，约 80～150 字。

### 工作贡献与责任担当

评级：`A · 优秀`

```
日常协作中，史千航在设计交付上承诺兑现情况较好。绩效系统相关页面的设计稿能按约定节点给出，对研发反馈的交互细节会跟进修改并闭环，很少出现遗漏后无人跟进的情况。遇到设计与实现不一致时，也会主动对齐方案而不是把问题悬空，整体责任担当表现稳定可靠。
```

### 协作沟通与价值观

评级：`A · 优秀`

```
跨角色协作时沟通比较顺畅：提设计评审会提前说明改动点，研发对稿时能耐心解释交互意图，也愿意根据实现成本调整方案。团队讨论里能尊重不同意见，不推诿、不甩锅，协作氛围正向。价值观层面与团队要求一致，愿意为共同交付目标让步和配合。
```

### 学习成长与潜力

评级：`A · 优秀`

```
学习意愿较强，能快速吸收新业务场景下的设计约束。绩效系统从零搭建阶段，对复杂表单与拆分布局等交互能主动查资料并对齐行业实践，也会把踩坑经验同步给组内同学。持续在设计工具与组件规范上保持更新，成长曲线清晰，后续承担更复杂产品模块的潜力较大。
```

若被评估人不是 UI/设计角色，按左侧「基本信息」职务改写示例内容，仍保持三段结构与 A 评级。

## 保存确认

点击「保存草稿」后：

- 表单控件短暂 `disabled` = 请求发送中
- 「保存草稿」恢复可用 = 请求结束
- 成功 toast：`360°评估草稿已保存`（可能很快消失）
- **强确认**：刷新后三项 `A · 优秀` 仍 checked，三段 textarea 有内容

若保存失败，toast 为错误文案；向用户报告错误并停在当前页，勿 unlock 后假装成功。

## 提交（仅用户明确要求时）

1. 确认三项评级已选（提交校验必填评级）
2. 点击「提交评估」
3. 成功后通常跳转 `/review-tasks`；toast 为「360°评估已提交」或「已重新提交并生效」

## 常见阻塞

| 现象 | 处理 |
|------|------|
| 「缺少 assignment_id」 | 向用户要完整 fill URL |
| 「任务尚未开放」 | 报告阶段未到，勿硬填 |
| 「无法加载 360°评估任务」 | 截图错误信息后停下 |
| 已生效只读 | 先点「编辑并重新提交」再填写 |
| 登录失效 | 停下请用户登录后重试 |
