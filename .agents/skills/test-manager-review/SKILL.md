# 测试上级评估

用 Cursor Browser 打开上级评估填写页，按以下顺序填写三个维度（分数 + 评语）并保存草稿。默认只保存草稿，不提交；仅当用户明确要求时才点「提交上级评估」。

多人造数、线上批量、随机 A/B 档分数 → 用 `batch-manager-review-api`（不走浏览器）。

## URL

优先使用用户给出的完整 URL。未给出时使用：

```
http://localhost:3001/review-tasks/fill?participant_id=<id>&type=MANAGER_REVIEW
```

必需参数：`type=MANAGER_REVIEW`、`participant_id`（上级评估按参与人加载，无 `assignment_id`）。

成功标志：页头为「上级评估」，右侧出现三个「分数（0–100）+ 评价」区与「保存草稿」「提交上级评估」。

## 操作流程

```
- [ ] 1. browser_tabs list；已有标签则先 browser_lock
- [ ] 2. browser_navigate 打开目标 URL（无标签时 navigate → lock）
- [ ] 3. browser_snapshot + screenshot 确认表单已加载（非「正在加载」）
- [ ] 4. 核心业绩：填分数 → 填写评语
- [ ] 5. 价值观：填分数 → 填写评语
- [ ] 6. 职业素养与潜力：填分数 → 填写评语
- [ ] 7. 点击「保存草稿」
- [ ] 8. 等待按钮从 disabled 恢复可用
- [ ] 9. 刷新页面，确认三项分数与三段评语回填
- [ ] 10. screenshot 后 browser_lock unlock，向用户汇报
```

分数输入用 `browser_fill` 写入 `aria-label` 为「*分数」的 textbox（placeholder `0–100`）。填入后 ScoreSelector 会显示命中等级按钮（如 `A · 优秀`），属正常反馈。

## 字段填写规范

三个维度结构相同：必填 `SCORE`（0–100）+ 长文本评语。以上级视角写可量化业绩与行为观察，评语约 80～120 字。

### 核心业绩

分数：`88`（命中 `A · 优秀`）

```
本周期作为 UI 设计师，在绩效系统多个核心页面的交互与视觉交付上表现稳定：设计稿节点可控，对研发反馈能及时闭环修改，少有遗漏悬空。复杂表单与拆分布局等难点能主动对齐方案，产出质量达到优秀档，能支撑业务按期推进。
```

### 价值观

分数：`90`（命中 `S · 卓越`）

```
跨角色协作态度正向：评审会提前说明改动意图，对研发实现成本能理性取舍，不推诿、不甩锅。团队讨论中尊重不同意见，愿意为共同交付目标配合调整。日常沟通坦诚、守约，价值观表现与团队要求高度一致。
```

### 职业素养与潜力

分数：`86`（命中 `A · 优秀`）

```
职业素养扎实：设计规范执行到位，交付物命名与标注清晰，便于研发对接。学习成长曲线清晰，能快速吸收新业务约束并把踩坑经验同步组内。后续可承担更复杂产品模块的主设职责，潜力达到优秀档。
```

若被评估人不是 UI/设计角色，按左侧「基本信息」职务改写示例内容，分数可保持 86/88/90 或微调，仍覆盖三维度。

若页面含「晋升评估」子表单（如晋升建议 SINGLE_SELECT），按可见选项填写后再保存；当前默认基线模板以三项 SCORE 为主。

## 保存确认

点击「保存草稿」后：

- 表单控件短暂 `disabled` = 请求发送中
- 「保存草稿」恢复可用 = 请求结束
- 成功 toast：`上级评估草稿已保存`（可能很快消失）
- **强确认**：刷新后三项分数仍为 88 / 90 / 86，三段 textarea 有内容

若保存失败，toast 为错误文案；向用户报告错误并停在当前页，勿 unlock 后假装成功。

## 提交（仅用户明确要求时）

1. 确认三项分数与必填评语已填（提交校验必填项）
2. 点击「提交上级评估」
3. 成功后通常跳转 `/review-tasks`；toast 为「上级评估已提交并完成系统计算」或「已重新提交并生效」

## 常见阻塞

| 现象 | 处理 |
|------|------|
| 「缺少 participant_id」 | 向用户要完整 fill URL |
| 「任务尚未开放」 | 报告阶段未到，勿硬填 |
| 「无法加载上级评估任务」 | 截图错误信息后停下 |
| 已生效只读 | 先点「编辑并重新提交」再填写 |
| 登录失效 | 停下请用户登录后重试 |
