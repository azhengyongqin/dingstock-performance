# 盯潮绩效系统大规模功能完善方案

> 状态：设计冻结，尚未实施  
> 决策日期：2026-07-14  
> 依据：`CONTEXT.md`、`docs/adr/0001`～`0060`、目标规则文档及当前代码实现  
> 冲突处理：具体 ADR 优先；同一主题下编号更高且明确声明取代关系的 ADR 优先

## 1. 目标与范围

本轮完善覆盖三个主域：

1. 模板管理：拆分配置模板与评估表单模板，提供发布版本、组合、校验和默认 D/M 模板。
2. 绩效周期：简化创建流程，重构周期与参与者状态，支持软截止、逐人校准、整体退回与最终归档。
3. 评估与结果：统一自评、360°、上级评估表单模型，建立可复算计分链、校准边界、结果版本和权限可见性。

目标规则来源：

- `docs/盯潮-绩效系统-绩效等级定义和计算方式.md`
- `docs/盯潮-绩效系统-评估维度规则说明.md`

本方案不在当前步骤修改 Prisma、数据库迁移、后端接口或前端页面。

## 2. 当前实现与主要差距

### 2.1 当前已有能力

- 后端已经具备模板、周期、参与者、评审员指派、自评、360°、上级评估、AI、校准、结果确认与申诉的基础服务。
- 前端已经具备模板设置、周期列表与编辑、自评、评审任务、团队评估、校准和结果页骨架。
- 周期创建已经存在模板复制与启动前检查的基础实现。
- 360°和上级评估已有草稿、提交及维度填写能力。

### 2.2 必须重构的差距

| 当前实现 | 目标状态 |
|---|---|
| 单一 `PerfTemplate` 同时承载规则、D/M 范围和表单 | 配置模板与评估表单模板分离，并分别版本化发布 |
| 自评、360°、上级评估使用三套数据模型 | 统一人工评估提交父模型与关系化评估项结果 |
| 维度评分主要保存在 `dimensionScores` JSON | 可计算结果关系化，支持查询、汇总和审计 |
| 上级评估手工选择 `initialLevel` | 系统按维度、权重、区间与约束计算阶段等级 |
| 360°评审员直接混合平均 | 先关系内平均，再对有效关系权重归一化 |
| 周期使用多个互斥业务阶段状态 | 周期只保留 DRAFT/SCHEDULED/ACTIVE/ARCHIVED |
| 参与者状态混入自评、AI、评审任务进度 | 参与者状态只表达结果主链，任务状态独立 |
| 截止时间可能终止任务或推进流程 | 填写截止时间只触发飞书通知 |
| 提交后锁定或依赖退回 | 校准前由原填写人直接修改、重新提交 |
| 校准只记录发生变化的调整 | 每个有结果参与者必须有 KEEP/ADJUST 明确决定 |
| 结果使用单一当前记录 | 结果、推送、确认按不可变结果版本关联 |
| AI 被建模为流程阶段 | AI 是异步参考信息，不阻塞任何流程 |

## 3. 目标领域模型

### 3.1 模板层

#### 配置模板

配置模板版本包含：

- 固定 S/A/B/C 评级定义、分数区间和映射分。
- 360°与上级评估的阶段结果模式。
- 维度约束规则与红线硬约束引用。
- 360°四类关系基础权重。
- 选定的已发布评估表单模板版本。
- 相对任务开始时间、填写提醒截止时间及飞书通知规则。

只有 Admin 可以创建、编辑草稿、发布和归档复用模板。HR只能查看并选用已发布版本。

#### 评估表单模板

每份表单模板绑定一个或多个受控职级前缀，并包含四类子表单：

- 员工自评。
- 360°评估。
- 上级评估。
- 晋升评估。

发布版本不可原地修改。配置模板必须绑定具体已发布版本，不能自动跟随最新版。

#### 发布校验

发布时至少校验：

- 职级前缀合法且配置模板内不重叠。
- 每个加权子表单的常规维度权重合计严格等于 100%。
- 每个加权子表单恰好有一个核心维度。
- 每个常规维度恰好有一个与阶段结果模式匹配的计分项。
- 评级区间连续覆盖 0～100，映射分落在所属区间。
- 关系权重四项均大于 0、最多两位小数且合计严格等于 100%。
- 晋升子表单包含员工和 Leader区段，不包含 360°评审员区段。

### 3.2 默认 D/M 表单

#### 员工自评

- 必填直接评级 S/A/B/C。
- 自评总结、半年度总结、下半年规划按目标规则初始化。
- 支持 Markdown、多行文本、附件和链接等受控组件。
- 启用晋升评估时，员工晋升材料随自评一起提交。

#### 上级评估

普通岗 D：

- 核心业绩 70%（核心维度）。
- 价值观 20%。
- 职业素养与潜力 10%。

管理岗 M：

- 核心业绩 50%（核心维度）。
- 管理绩效 50%。

默认采用加权评分，每个常规维度只有一个 0～100 分计分项。

#### 360°评估

普通岗 D：

- 工作贡献与责任担当 35%（核心维度）。
- 协作沟通与价值观 45%。
- 学习成长与潜力 20%。

管理岗 M：

- 结果推动与责任担当 40%（核心维度）。
- 协作沟通与组织影响 35%。
- 领导力与价值观 25%。

默认采用加权评级。系统不提供“了解不足、无法评价”操作；未提交答卷保持草稿且不计分。

### 3.3 周期与任务层

#### 周期状态

```text
DRAFT <-> SCHEDULED -> ACTIVE -> ARCHIVED
   ^          ^           |
   +----------+-----------+
      仅 Admin 可从 ACTIVE 退回
```

- `DRAFT`：配置和参与者草稿。
- `SCHEDULED`：等待计划启动。
- `ACTIVE`：参与者按个人进度评估、校准、确认和申诉。
- `ARCHIVED`：永久不可逆终态。

到达计划启动时间时自动检查并启动。校验失败保持 `SCHEDULED` 并通知 HR/Admin。

#### 任务时间

- 开始时间是硬开放门槛。
- 填写提醒截止时间只触发飞书通知，不关闭任务、不改变状态。
- 任务开放后持续可写，直到参与者校准锁定、无结果、中途退出、周期退回暂停或周期归档。

#### 周期退回

- 只有 Admin 可以把 `ACTIVE` 退回 `SCHEDULED` 或 `DRAFT`。
- 不受已有校准、推送、确认或申诉限制。
- 现有校准、结果版本、确认和申诉标记失效，历史永久保留。
- 已校准参与者解除评估锁定，重新激活后必须重新校准。
- 已推送结果的员工收到“周期退回、原结果失效”飞书通知。
- `ARCHIVED` 不能退回。

### 3.4 参与者结果主链

参与者状态：

```text
ACTIVE -> CALIBRATED -> RESULT_PUBLISHED -> CONFIRMED
                               |               ^
                               v               |
                           APPEALING -> RECONFIRMING

ACTIVE -> NO_RESULT
ACTIVE -> WITHDRAWN
```

- 自评、360°、上级评估、AI 任务进度不进入参与者主状态。
- `NO_RESULT` 仅用于员工始终未提交必需自评，由 HR/Admin显式设置。
- 上级评估缺失不能取消员工结果，必须催办或更换考核 Leader。
- 周期归档后参与者随周期统一只读，不再设置参与者 `ARCHIVED`。

### 3.5 人工评估提交

- 自评、360°和上级评估共享统一提交模型。
- 草稿允许不完整，正式提交必须通过完整性校验。
- 每名填写人只保留一份当前答卷，不保留答卷历史版本。
- 已有成功提交再次编辑时，上次提交继续参与计算；新修改保存在临时草稿。
- 重新提交成功后原子替换当前有效结果并触发重算。
- 首次校准前可持续修改；首次校准成功后按参与者锁定全部人工答卷。
- 周期整体退回是解除评估锁定的唯一例外。

表单结构变化影响完整性或计算语义时，必须先退回草稿；受影响答卷转为草稿并由原填写人重新提交，兼容旧值自动预填。

## 4. 计算规则

### 4.1 通用规则

- 评级模式保存原始评级和评级映射分。
- 评分模式保存原始分数，计算分等于原始分数。
- 输入范围 0～100，最多两位小数。
- 中间计算使用精确十进制且不提前舍入。
- 阶段综合分四舍五入至两位小数后映射等级。
- 阶段结果保存综合分、初始等级、约束原因和阶段等级。

### 4.2 默认映射

- S：95。
- A：85。
- B：70。
- C：50。

Admin可在配置模板中调整，但映射分必须落在所属等级区间。

### 4.3 360°关系权重

默认基础权重：

- 组织负责人 30%。
- 项目负责人 30%。
- 同部门同事 25%。
- 跨部门协作方 15%。

计算顺序：

1. 每名已提交评审员的原始评级或评分转换为各维度计算分。
2. 同一关系内按维度做算术平均。
3. 排除没有有效提交的关系。
4. 对剩余关系的基础权重重新归一化。
5. 得到每个维度的关系加权分。
6. 按维度权重计算阶段综合分。
7. 映射初始等级并应用核心维度约束。

未提交草稿不计分；存在尚未重新提交的更新草稿时，继续使用上一次成功提交。全部无有效提交时产生无数据阶段结果。

### 4.4 最终结果权威链

- 自评阶段等级：员工必填直接评级，仅作参考。
- 360°阶段等级：加权结果，仅作参考。
- AI 阶段等级：异步参考，不阻塞流程。
- 上级评估阶段等级：校准前权威等级。
- 校准结果：最终人工权威边界。
- 有效红线认定：强制最终等级为 C。

配置重算只更新阶段结果，不自动穿透已有校准结果。需要改变最终结果时必须显式重新校准。

## 5. 校准与结果

### 5.1 校准前置条件

- 员工自评已正式提交。
- 上级评估已正式提交并产生阶段等级。
- 360°允许部分提交或完全无数据。
- AI允许生成中、失败或缺失。

### 5.2 权限

- 当前考核 Leader：仅自己责任范围。
- HR：仅授权组织范围。
- Admin：全局。

Leader变化后，未来查看和重新校准权限转给新 Leader，历史记录保留原操作人。

### 5.3 决定与并发

- 每个有结果参与者必须明确提交 `KEEP` 或 `ADJUST`。
- `ADJUST` 必须填写原因。
- 校准记录追加保存，不覆盖历史。
- 提交校准时校验最新人工评估修订、红线状态和最新校准标识。
- 发现并发变化时拒绝提交并要求刷新，不采用最后写入覆盖。

### 5.4 结果版本

- 结果推送、员工确认和申诉都关联不可变结果版本。
- 重新校准只有在等级变化时生成新结果版本并重新确认。
- 等级未变化的再次 KEEP 不打扰员工。
- 周期退回时当前结果链失效并通知员工。

### 5.5 员工可见性

员工可见：

- 最终等级及等级说明。
- 上级评估综合分、维度分和 Leader评语。
- 自己的自评内容与自评等级。
- 配置允许时的晋升结论。
- 结果版本、确认状态和改判前后等级。

员工不可见：

- 360°评审员身份、个人评分、评语、关系汇总、阶段分和阶段等级。
- AI 报告与 AI 建议等级。
- 内部校准讨论与敏感校准备注。

## 6. 建议数据库模型

以下为目标模型建议，最终以实施时修改后的 `backend/prisma/schema.prisma` 为准。

### 6.1 模板与周期配置

| 建议模型 | 责任 |
|---|---|
| `PerfConfigTemplate` | 配置模板稳定标识 |
| `PerfConfigTemplateVersion` | 草稿/发布/归档版本与规则快照 |
| `PerfFormTemplate` | 表单模板稳定标识与职级前缀 |
| `PerfFormTemplateVersion` | 表单发布版本 |
| `PerfFormSubform` | SELF/PEER/MANAGER/PROMOTION 子表单 |
| `PerfFormDimension` | 维度、权重、核心标记、排序 |
| `PerfFormItem` | 受控组件、必填与限制配置 |
| `PerfConfigFormBinding` | 配置模板版本绑定表单模板版本 |
| `PerfCycleConfigVersion` | 周期配置版本与变更原因 |
| `PerfCycleSchedule` | 开始时间、提醒时间与开放事件 |
| `PerfCycleForm/Dimension/Item` | 周期独立表单快照 |

评级、约束和通知策略可使用受校验 JSON 快照；经常关联、过滤和汇总的模板绑定、维度、评估项必须关系化。

### 6.2 评估与计算

| 建议模型 | 责任 |
|---|---|
| `PerfReviewerAssignment` | 评审员、关系、替换状态与任务开放 |
| `PerfEvaluationSubmission` | 三类人工答卷的当前有效提交元数据 |
| `PerfEvaluationItemResult` | 当前有效评估项值、原始值和计算分 |
| `PerfStageResult` | 阶段综合分、等级、约束原因和配置版本 |
| `PerfStageDimensionResult` | 各维度最终分与等级 |
| `PerfPeerRelationAggregate` | 关系内均分和归一化权重 |

临时编辑草稿可保存在提交记录的结构化 JSON 草稿载荷中；正式可计算结果关系化保存。重新提交原子替换当前有效子记录，不建立答卷历史版本。

### 6.3 校准、结果与退回

| 建议模型 | 责任 |
|---|---|
| `PerfCalibrationDecision` | KEEP/ADJUST、输入修订、操作人与失效状态 |
| `PerfRedLineFinding` | 红线确认、证据、撤销与审计 |
| `PerfResultVersion` | 不可变员工结果版本及有效状态 |
| `PerfResultConfirmation` | 本人确认或视同确认及对应结果版本 |
| `PerfAppeal` | 一次申诉、处理与结果版本关联 |
| `PerfCycleRollback` | 超级管理员周期退回及影响摘要 |

所有常用当前记录路径需要唯一约束和索引；所有失效链必须保留失效原因、操作者和时间。

## 7. 前端功能结构

### 7.1 模板管理

配置模板页面：

- 版本列表与状态。
- 评级区间、映射分和阶段模式。
- 约束规则。
- 360°关系权重配置页面。
- D/M 表单模板版本选择与前缀覆盖预览。
- 日程与飞书通知配置。
- 发布校验结果与版本差异。

评估表单模板页面：

- 职级前缀选择。
- 自评、360°、上级评估、晋升评估四个子表单 Tab。
- 维度、权重、核心维度和受控评估项编辑。
- D/M 默认模板初始化。
- 填写预览与只读历史版本。

### 7.2 周期创建

固定四步：

1. 周期名称、配置模板版本、计划启动时间。
2. 参与者、职级前缀匹配、晋升标记。
3. 任务开始时间和提醒日程预览。
4. 启动检查、保存草稿、设为待启动或立即启动。

复杂规则只在“高级配置”中修改，不进入默认创建路径。

### 7.3 填写与校准

- 三类人工表单共用动态渲染器、自动草稿与提交校验。
- 已提交答卷支持继续编辑，清晰显示“当前有效提交”和“未提交修改”。
- 校准页展示自评、360°关系汇总、上级评估、AI、红线和差异提示。
- 并发冲突显示变化来源并要求刷新。
- 周期退回提供影响预览、二次确认和员工通知结果。

## 8. 实施顺序

### 阶段一：领域枚举、数据库与计算内核

- 新增目标模型、约束和索引。
- 建立统一精确十进制计算引擎。
- 覆盖评级、评分、约束、关系归一化和无数据测试。
- 暂不切换现有页面。

成功标志：给定相同规则快照和输入，服务端可稳定重放全部阶段计算。

### 阶段二：模板管理

- 实现两类模板与不可变发布版本。
- 初始化 D/M 四子表单和默认关系权重。
- 完成发布校验、权限和版本历史。

成功标志：Admin可发布一份完整配置模板，HR可只读选择，非法权重或不兼容表单无法发布。

### 阶段三：周期创建与生命周期

- 实现四步创建流程。
- 重构周期与参与者状态。
- 实现自动启动、任务开放、软截止通知和归档检查。
- 实现超级管理员整体退回。

成功标志：同一 ACTIVE 周期内可同时存在评估中、已校准和待确认参与者，互不错误阻塞。

### 阶段四：统一评估表单

- 上线统一提交与动态表单渲染。
- 支持当前有效提交、临时更新草稿和重新提交。
- 上线 D/M 自评、360°、上级和晋升表单。
- 实现 360°关系汇总。

成功标志：D/M 完整评估链可计算，360°缺少关系时正确归一化，未提交草稿永不进入计算。

### 阶段五：校准、结果版本与可见性

- 实现逐人锁定、乐观并发和 KEEP/ADJUST。
- 实现结果版本、重新推送、确认与申诉关联。
- 实现员工专用结果响应和敏感字段隔离。

成功标志：并发校准不会静默覆盖，重新校准只在等级变化时重开员工确认。

### 阶段六：迁移与切换

- 迁移旧模板、周期快照、三类评估和 JSON 维度评分。
- 对旧人工初评等级与新计算等级生成差异报告。
- 先影子计算，再按周期切换读取路径。
- 完成回滚预案后移除旧写入路径。

成功标志：历史数据可查，进行中周期无丢失，切换前后结果差异均有解释。

## 9. 核心验收标准

### 模板

- 未发布模板不能创建周期。
- 发布版本不可原地修改。
- 职级前缀必须唯一匹配。
- D/M 默认表单、晋升子表单和360°关系权重完整存在。

### 计算

- 评级与评分都保存原始值和计算分。
- 有效关系缺失时权重准确归一化。
- 中间结果不提前舍入，最终综合分两位小数。
- 红线强制 C 不可被校准或申诉直接覆盖。

### 流程

- 截止时间只发通知，不关闭任务。
- 自评与上级评估未提交时不能校准。
- 360°完全无提交时得到无数据结果。
- 首次校准后逐人锁定，周期退回后整体解锁并失效结果链。
- 已归档周期不可修改或退回。

### 权限与隐私

- Admin管理复用模板和高风险周期退回。
- HR按组织范围运营周期。
- Leader只访问当前责任范围。
- 员工接口无法获取360°、AI和内部校准敏感字段。

### 审计

- 模板发布、周期配置修改、Leader更换、校准、红线、结果推送、确认、申诉、无结果设置、周期退回和归档均可追溯。
- 不保留答卷内容历史版本，但保留必要操作事件和当前有效内容。

## 10. 实施前置提醒

- 修改 Prisma 时，必须同步更新 `docs/数据库表结构.md` 并提供迁移脚本。
- 状态枚举变化影响后端守卫、看板查询、前端状态标签和历史数据迁移，应作为同一批次处理。
- 计算引擎必须先建立单元测试向量，再接入表单提交和校准。
- 旧产品详细设计文档中关于 AI 独立阶段、自评退回、360°参与晋升、硬截止和人工初评等级的描述已被本方案及 ADR取代，实施时应同步清理。
