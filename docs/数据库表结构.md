# 数据库表结构

本文档与 `backend/prisma/schema.prisma` 同步维护。当前数据库使用 PostgreSQL，所有业务表位于 `performance` schema。

## 维护规则

- 修改数据库结构时，必须同时更新 `backend/prisma/schema.prisma` 和本文档。
- Prisma model 使用 `@@map` 映射到 snake_case 复数表名。
- 所有表显式声明 `@@schema("performance")`。
- 飞书同步表尽量保持 SDK 原始字段形态；嵌套对象使用 JSONB，标量数组使用 PostgreSQL 数组。
- 现有飞书同步表保持当前 snake_case Prisma 字段风格；新增项目自有业务表默认使用 camelCase Prisma 字段，并通过 `@map` 映射到 snake_case 数据库列。
- 新增项目自有业务表默认包含基础字段：`id`（自增主键）、`created_at`、`updated_at`、`deleted_at`（支持软删除时使用）。
- 新增项目自有业务表必须设计显式外键、明确 `onDelete` 策略和必要索引。
- 每张表的字段说明表必须列出 `schema.prisma` 中的全部字段，包括基础字段、外键标量字段、时间字段和软删除字段；禁止只写“主要字段”后省略字段。

## 新业务表基础字段规范

| Prisma 字段 | 数据库列 | 类型建议 | 说明 |
|---|---|---|---|
| `id` | `id` | `Int @id @default(autoincrement())` | 内部自增主键，不承载业务含义 |
| `createdAt` | `created_at` | `DateTime @default(now())` | 创建时间 |
| `updatedAt` | `updated_at` | `DateTime @updatedAt` | 最近更新时间；append-only 表可不加 |
| `deletedAt` | `deleted_at` | `DateTime?` | 软删除时间；不可变历史表不使用 |

Prisma 示例：

```ts
id        Int       @id @default(autoincrement())
createdAt DateTime  @default(now()) @map("created_at")
updatedAt DateTime  @updatedAt @map("updated_at")
deletedAt DateTime? @map("deleted_at")
```

### 业务域建模总原则

- **员工引用不建物理外键**：业务表统一以飞书 `open_id`（String）引用员工，展示层 join `lark_users`。原因：同步表全量重建/员工离职删除时，绩效历史必须独立存活。
- **快照字段**：`perf_participants` 在周期启动时写入部门/职级/Leader 快照，组织变动不影响历史归属（产品 §6.1.5、§6.6.4）。
- **JSONB 使用范围**：动态表单 schema、维度评分明细、AI 输出、时间窗口、催办规则、附件列表等不需要 SQL 过滤/join 的柔性结构。
- **状态字段全部使用 Prisma enum**，与后端状态机（研发文档 §8.2）一一对应。
- **append-only 表**（`perf_calibrations`、`audit_logs`）：无 `updated_at`/`deleted_at`，业务代码禁止 UPDATE/DELETE。
- **等级字段存 `String`**（如 `initial_level` / `final_level` / `before_level` / `after_level`）：等级集合（S/A/B/C/D 等）由各周期评分规则 `levels` 配置决定，不做全局枚举。
- **岗位差异化维度集**：同一周期内不同岗位类型适用不同的维度集和权重（如员工岗 D：核心业绩 70/职业素养与潜力 10/价值观 20；管理岗 M：核心业绩 50/管理绩效 50），通过 `perf_dimensions.applicable_scope` 圈定各维度适用人群；权重校验按适用分组分别合计 = 100，而非全周期合计。
- **归档不可变**：`perf_results.archived_at` 非空后禁止 UPDATE；任何调整必须经校准记录/操作日志产生新记录。
- **模板 + 创建时复制**：评分规则与维度集的复用走 `perf_templates` / `perf_template_dimensions` 模板表；创建周期时复制到 `perf_scoring_rules` / `perf_dimensions` 生成该周期的快照实例。之后改模板不影响已创建周期，周期内微调也不回写模板；`perf_cycles.template_id` 仅做来源溯源。

## 业务域枚举

均位于 `performance` schema，与后端状态机一一对应。

| 枚举 | 取值 | 用途 |
|---|---|---|
| `PerfCycleType` | `SEMI_ANNUAL`（半年度） / `QUARTERLY`（季度（预留）） / `ANNUAL`（年度（预留）） | 周期类型；首期只用半年度，季度/年度预留 |
| `PerfCycleStatus` | `DRAFT`（草稿：唯一允许删除周期的状态） / `PENDING`（待启动） / `SELF_REVIEW`（自评中） / `REVIEWING`（评审中） / `AI_ANALYZING`（AI 分析中） / `CALIBRATING`（校准中） / `CONFIRMING`（结果确认中） / `ARCHIVED`（已归档） | 周期状态机（研发文档 §8.2）：DRAFT → PENDING → SELF_REVIEW → REVIEWING → AI_ANALYZING → CALIBRATING → CONFIRMING → ARCHIVED |
| `PerfParticipantStatus` | `PENDING_SELF_REVIEW`（待自评） / `SELF_SUBMITTED`（自评已提交） / `RETURNED`（自评被退回（回退分支）） / `REVIEWED`（360°+上级评估完成） / `AI_DONE`（AI 分析完成） / `CALIBRATED`（已校准） / `RESULT_PUSHED`（结果已推送员工） / `CONFIRMED`（员工已确认） / `APPEALING`（申诉中） / `RE_CONFIRMING`（申诉处理后再次确认中） / `ARCHIVED`（已归档） | 参与者状态机（研发文档 §8.2），周期整体状态按参与者聚合推进 |
