# 数据库表结构

本文档与 `backend/prisma/schema.prisma` 同步维护。当前数据库使用 PostgreSQL，所有业务表位于 `performance` schema。

## 维护规则

- 修改数据库结构时，必须同时更新 `backend/prisma/schema.prisma` 和本文档。
- Prisma model 使用 `@@map` 映射到 snake_case 复数表名。
- 所有表显式声明 `@@schema("performance")`。
- 飞书同步表尽量保持 SDK 原始字段形态；嵌套对象使用 JSONB，标量数组使用 PostgreSQL 数组。
- 现有飞书同步表保持当前 snake_case Prisma 字段风格；新增项目自有业务表默认使用 camelCase Prisma 字段，并通过 `@map` 映射到 snake_case 数据库列。
- 新增项目自有业务表默认包含基础字段：`id`（自增主键）、`created_at`、`updated_at`、`deleted_at`（支持软删除时使用）。
- 新增项目自有业务表必须设计显式外键、明确 `onDelete` 策略和必要索引。
- 每张表的字段说明表必须列出 `schema.prisma` 中的全部字段，包括基础字段、外键标量字段、时间字段和软删除字段；禁止只写“主要字段”后省略字段。

## 新业务表基础字段规范

| Prisma 字段 | 数据库列 | 类型建议 | 说明 |
|---|---|---|---|
| `id` | `id` | `Int @id @default(autoincrement())` | 内部自增主键，不承载业务含义 |
| `createdAt` | `created_at` | `DateTime @default(now())` | 创建时间 |
| `updatedAt` | `updated_at` | `DateTime @updatedAt` | 最近更新时间；append-only 表可不加 |
| `deletedAt` | `deleted_at` | `DateTime?` | 软删除时间；不可变历史表不使用 |

Prisma 示例：

```ts
id        Int       @id @default(autoincrement())
createdAt DateTime  @default(now()) @map("created_at")
updatedAt DateTime  @updatedAt @map("updated_at")
deletedAt DateTime? @map("deleted_at")
```

### 业务域建模总原则

- **员工引用不建物理外键**：业务表统一以飞书 `open_id`（String）引用员工，展示层 join `lark_users`。原因：同步表全量重建/员工离职删除时，绩效历史必须独立存活。
- **周期配置快照**：新版周期从已发布配置模板复制 `perf_cycle_config_versions` 与 D/M `perf_cycle_form_snapshots`，之后模板发布、归档或修改其他草稿均不影响周期。参与人加入时先固化职级前缀和准确表单绑定，启动时再复核部门、完整职级与 Leader 快照。
- **JSONB 使用范围**：动态表单 schema、维度评分明细、AI 输出、时间窗口、催办规则、附件列表等不需要 SQL 过滤/join 的柔性结构。
- **状态字段全部使用 Prisma enum**，与后端状态机（研发文档 §8.2）一一对应。
- **append-only / 不可变表**（`perf_cycle_rollbacks`、`perf_calibrations`、`perf_red_line_findings`、`perf_result_versions`、`perf_legacy_promotion_archives`、`audit_logs`）：无 `updated_at`/`deleted_at`；周期整体退回只允许给校准和结果版本一次性补充失效元数据，结果版本还允许一次性写入替代时间或确认信息，其他业务快照禁止 UPDATE，全部禁止 DELETE。
- **评级字段**：维度原始评级和计算链使用 `PerfRatingSymbol`（S/A/B/C），不接受任意字符串。
- **岗位差异化维度集**：D/M 维度、权重和受众固化在已发布 `perf_form_template_versions`，周期启动后使用 `perf_cycle_form_snapshots.content`；SELF / PEER / MANAGER 全部读取维度作答、字段作答与阶段维度结果。
- **归档不可变**：周期归档写入 `perf_cycle_archives` 事实；结果历史始终保存在 append-only 的 `perf_result_versions`，任何调整必须生成新版本。
- **评估表单版本不可变**：只有 `DRAFT` 版本可修改 SELF / PEER / MANAGER 子表单、维度和字段。数据库触发器会拒绝对 `PUBLISHED` / `ARCHIVED` 版本及其层级的改写；旧 PROMOTION 内容仅在既有版本中只读保留。
- **配置模板版本不可变**：`perf_config_template_versions` 精确绑定已发布 D/M 表单版本；只有草稿可改规则和绑定，发布后只允许转为归档。评级、相对日程和通知规则使用受服务层类型校验的 JSONB 快照，表单绑定使用关系表。
- **最终切换边界**：contract migration 后只保留版本化模板、周期快照、统一提交、维度/字段作答、阶段结果与结果版本；旧评估项、旧结果、阶段模式、双套约束和迁移账本均已删除。旧晋升答案只有在 `20260720192500_archive_legacy_promotion_answers` 于 contract 前完成时，才可证明已完整写入 `perf_legacy_promotion_archives`。若环境曾先执行 contract，结果快照只能补回只读展示投影，无法恢复未进入快照的旧答案；存在任何旧晋升证据时，后续 completeness gate 会阻断部署，必须从 contract 前数据库备份恢复并按 migration 顺序重跑，禁止把补录投影视为完整归档。

用途：版本化配置模板的稳定标识。可复用的名称、评级、计算模式、约束、关系权重、表单组合、相对日程和通知策略都属于具体版本，不放在稳定标识上。

### 字段说明

| Prisma 字段 | 数据库列 | 类型 | 约束/默认值 | 说明 |
|---|---|---|---|---|
| `id` | `id` | `Int` | 主键；默认 autoincrement() | 内部自增主键 |
| `systemKey` | `system_key` | `String?` | 唯一；可空；非空时不得为空白字符串 | 内置配置模板幂等键；管理员自建模板为空 |
| `createdByOpenId` | `created_by_open_id` | `String` | 非空 | 创建人飞书 open_id；不建到同步表的物理外键，确保历史不受通讯录清理影响 |
| `createdAt` | `created_at` | `DateTime` | 默认 now() | 创建时间 |
| `updatedAt` | `updated_at` | `DateTime` | @updatedAt | 最近更新时间 |

### 关系

| Prisma 关系字段 | 类型 | 说明 |
|---|---|---|
| `versions` | `PerfConfigTemplateVersion[]` | 所有版本；版本外键 `onDelete: Restrict`，存在版本时禁止删除稳定模板 |

### 索引与唯一约束

| 类型 | 字段 | 说明 |
|---|---|---|
| 唯一约束 | `systemKey` | 支持内置配置模板幂等初始化；PostgreSQL 允许多个 `NULL` |

## perf_config_template_versions

Prisma model：`PerfConfigTemplateVersion`

用途：配置模板版本聚合根，统一保存固定等级、360°关系权重、相对日程和通知预设。草稿可不完整并自由编辑；发布后内容不可变，且必须准确绑定已发布的 D/M 评估表单版本。人工评估的混合计分方式属于各维度，固定约束由统一计算引擎执行。

### 字段说明

| Prisma 字段 | 数据库列 | 类型 | 约束/默认值 | 说明 |
|---|---|---|---|---|
| `id` | `id` | `Int` | 主键；默认 autoincrement() | 内部自增主键 |
| `templateId` | `template_id` | `Int` | 外键 | 所属稳定配置模板 |
| `version` | `version` | `Int` | 必须大于 0；与 `templateId` 联合唯一 | 模板内递增版本号，从 1 开始 |
| `status` | `status` | `PerfConfigTemplateVersionStatus` | 默认 DRAFT | DRAFT / PUBLISHED / ARCHIVED；数据库触发器控制可写边界 |
| `name` | `name` | `String` | 非空且不得为空白字符串 | 版本名称，发布后不可变 |
| `description` | `description` | `String?` | - | 版本说明 |
| `sourceVersionId` | `source_version_id` | `Int?` | 自关联外键；不得等于本记录 `id` | 复制生成草稿的来源版本；首次创建为空，重新指定时必须是同一模板的已发布版本 |
| `ratings` | `ratings` | `Json` | 默认 `[]`；必须为 JSON array | 固定 S/A/B/C 顺序的名称、说明、连续分数区间和映射分；发布时由服务层验证完整性 |
| `orgOwnerWeight` | `org_owner_weight` | `Decimal @db.Decimal(5, 2)` | 默认 30；0～100 | 组织负责人基础权重；发布时必须大于 0 |
| `projectOwnerWeight` | `project_owner_weight` | `Decimal @db.Decimal(5, 2)` | 默认 30；0～100 | 项目负责人基础权重；发布时必须大于 0 |
| `peerWeight` | `peer_weight` | `Decimal @db.Decimal(5, 2)` | 默认 25；0～100 | 同部门同事基础权重；发布时必须大于 0 |
| `crossDeptWeight` | `cross_dept_weight` | `Decimal @db.Decimal(5, 2)` | 默认 15；0～100 | 跨部门协作方基础权重；发布时必须大于 0 |
| `schedulePreset` | `schedule_preset` | `Json` | 默认 `{}`；必须为 JSON object | 相对计划启动时间的 SELF / PEER / MANAGER 开放偏移、填写提醒偏移和阶段重叠策略，不保存绝对日期 |
| `notificationRules` | `notification_rules` | `Json` | 默认 `{}`；必须为 JSON object | 任务开放与填写提醒的飞书通知开关、受控接收人、抄送人和提醒频率；不代表实际通知事件 |
| `createdByOpenId` | `created_by_open_id` | `String` | 非空 | 创建版本的管理员 open_id |
| `updatedByOpenId` | `updated_by_open_id` | `String` | 非空 | 最近修改版本的管理员 open_id |
| `publishedByOpenId` | `published_by_open_id` | `String?` | PUBLISHED / ARCHIVED 时必填 | 发布人 open_id |
| `publishedAt` | `published_at` | `DateTime?` | PUBLISHED / ARCHIVED 时必填 | 发布时间 |
| `archivedByOpenId` | `archived_by_open_id` | `String?` | ARCHIVED 时必填 | 归档人 open_id |
| `archivedAt` | `archived_at` | `DateTime?` | ARCHIVED 时必填 | 归档时间 |
| `createdAt` | `created_at` | `DateTime` | 默认 now() | 创建时间 |
| `updatedAt` | `updated_at` | `DateTime` | @updatedAt | 最近更新时间 |

`ratings` 预期形状：

```json
[
  {
    "symbol": "S",
    "name": "卓越",
    "description": "显著超出预期",
    "minScore": "90",
    "maxScore": "100",
    "mappingScore": "95"
  }
]
```

数组必须且只能包含 S、A、B、C，符号和展示顺序不可改；分数使用十进制字符串，避免 JSON number 在跨语言复制时产生精度歧义。除最高档外区间右开，S 档右闭。

人工评估固定使用统一约束：核心维度 C 强制 C、核心维度 B 最高 B、任一维度 C 最高 B；已确认红线继续独立强制 C。数据库不保存可配置的双套约束档位。

`schedulePreset` 预期形状：

```json
{
  "allowStageOverlap": true,
  "stages": [
    { "stage": "SELF", "startOffsetMinutes": 0, "reminderDeadlineOffsetMinutes": 10080 },
    { "stage": "PEER", "startOffsetMinutes": 10080, "reminderDeadlineOffsetMinutes": 20160 },
    { "stage": "MANAGER", "startOffsetMinutes": 20160, "reminderDeadlineOffsetMinutes": 30240 }
  ]
}
```

填写提醒时间是软截止，仅触发通知；真正任务开放由开始偏移生成的绝对时间控制。AI 不属于人工填写任务，不放入任务日程。

`notificationRules` 预期形状：

```json
{
  "stages": [
    {
      "stage": "SELF",
      "taskOpened": {
        "enabled": true,
        "recipient": "ASSIGNEE",
        "ccLeader": true,
        "ccHr": false
      },
      "reminder": {
        "enabled": true,
        "recipient": "ASSIGNEE",
        "ccLeader": true,
        "ccHr": false,
        "frequency": { "type": "EVERY_N_DAYS_AFTER_DEADLINE", "intervalDays": 1 }
      }
    }
  ]
}
```

每个人工填写阶段必须且只能配置一条规则；接收人固定为任务执行人，抄送开关与提醒频率均由服务层白名单校验。`intervalDays` 只允许用于 `EVERY_N_DAYS_AFTER_DEADLINE`。模板只保存策略；实际通知事件、重试状态和发送结果仍写入 `perf_notifications`。

### 关系

| Prisma 关系字段 | 类型 | 说明 |
|---|---|---|
| `template` | `PerfConfigTemplate` | `onDelete: Restrict`；存在版本时禁止删除稳定配置模板 |
| `sourceVersion` | `PerfConfigTemplateVersion?` | `onDelete: Restrict`；有派生版本时禁止删除来源版本 |
| `derivedVersions` | `PerfConfigTemplateVersion[]` | 从本版本复制生成的后续版本 |
| `formBindings` | `PerfConfigFormBinding[]` | `onDelete: Cascade`；仅删除草稿版本时级联删除表单绑定 |
| `cycleVersions` | `PerfCycleConfigVersion[]` | 周期配置快照的来源引用；`onDelete: Restrict`，已创建周期后保留来源版本 |

### 索引、唯一约束与不可变规则

| 类型 | 字段 | 说明 |
|---|---|---|
| 唯一约束 | `templateId, version` | 同一稳定配置模板的版本号不重复 |
| 索引 | `templateId, status` | 查询模板版本历史和草稿 |
| 索引 | `status` | HR 查询可选的已发布版本 |
| 索引 | `sourceVersionId` | 查询版本派生关系 |
| CHECK | JSON 顶层类型 | `ratings` 为数组；`schedulePreset`、`notificationRules` 为对象 |
| CHECK | 四类关系权重 | 草稿可保存 0～100；PUBLISHED / ARCHIVED 每项必须大于 0 且严格合计 100 |
| CHECK | `status` 与生命周期元数据 | DRAFT 不得有发布/归档元数据；PUBLISHED 必须有发布人和时间；ARCHIVED 必须同时有发布与归档元数据 |
| 触发器 | 版本 INSERT / UPDATE / DELETE | 只能创建草稿、只有草稿可删除；模板归属、版本号和创建身份始终不可变；DRAFT 仅能发布；PUBLISHED 仅能在不改内容时归档；ARCHIVED 禁止修改 |
| 触发器 | DRAFT → PUBLISHED | 必须恰好绑定一份 D 表单和一份 M 表单；以共享行锁串行化表单归档并确认发布瞬间两者都仍为 PUBLISHED；再次检查四类关系权重 |

`Decimal(5,2)`保证数据库最终只保存两位小数，但 PostgreSQL 对更多小数可能先舍入；API 必须在写入前拒绝超过两位小数的关系权重。

## perf_config_form_bindings

Prisma model：`PerfConfigFormBinding`

用途：把配置模板版本准确绑定到具体的已发布评估表单版本。当前配置发布要求 D、M 各且仅一条，不跟随评估表单的后续新版本。

### 字段说明

| Prisma 字段 | 数据库列 | 类型 | 约束/默认值 | 说明 |
|---|---|---|---|---|
| `id` | `id` | `Int` | 主键；默认 autoincrement() | 内部自增主键 |
| `configVersionId` | `config_version_id` | `Int` | 外键；与 `jobLevelPrefix` 联合唯一 | 所属配置模板版本 |
| `formTemplateVersionId` | `form_template_version_id` | `Int` | 复合外键 | 被绑定的评估表单模板版本 ID |
| `jobLevelPrefix` | `job_level_prefix` | `PerfJobLevelPrefix` | 复合外键；与 `configVersionId` 联合唯一 | 冗余 D/M 前缀；必须与被绑定表单版本的前缀一致 |
| `createdAt` | `created_at` | `DateTime` | 默认 now() | 创建时间 |
| `updatedAt` | `updated_at` | `DateTime` | @updatedAt | 最近更新时间；只有父配置版本为草稿时可更新 |

### 关系

| Prisma 关系字段 | 类型 | 说明 |
|---|---|---|
| `configVersion` | `PerfConfigTemplateVersion` | `onDelete: Cascade`；绑定无法脱离配置版本存在，且只有草稿版本允许删除 |
| `formTemplateVersion` | `PerfFormTemplateVersion` | 复合外键 `[formTemplateVersionId, jobLevelPrefix]`；`onDelete: Restrict` 保留历史来源 |

### 索引、唯一约束与不可变规则

| 类型 | 字段 | 说明 |
|---|---|---|
| 唯一约束 | `configVersionId, jobLevelPrefix` | 同一配置版本每个职级前缀至多绑定一份表单版本 |
| 索引 | `formTemplateVersionId` | 反查评估表单版本被哪些配置版本引用 |
| 复合外键 | `formTemplateVersionId, jobLevelPrefix` | 引用 `perf_form_template_versions(id, job_level_prefix)`，数据库直接阻止伪造或错配前缀 |
| 触发器 | INSERT / UPDATE | 父配置版本必须为 DRAFT，目标评估表单版本在绑定当时必须为 PUBLISHED |
| 触发器 | UPDATE / DELETE | 新旧父链都必须属于 DRAFT，防止移动已发布绑定绕过不可变保护 |

评估表单版本之后归档不会删除既有绑定。已有已发布配置保留完整历史来源；尚未发布的配置草稿会在发布时因来源已归档而被拒绝，管理员需改绑新的已发布版本。

## perf_cycle_config_versions

Prisma model：`PerfCycleConfigVersion`

用途：保存绩效周期独立的计算配置、相对日程和通知规则快照。创建周期时从一个已发布配置模板版本完整复制为版本 1；DRAFT / SCHEDULED 阶段可按权限调整而不回写模板，ACTIVE 后旧版本不可原地修改。后续活动周期的非结构性规则调整通过追加新版本实现。

### 字段说明

| Prisma 字段 | 数据库列 | 类型 | 约束/默认值 | 说明 |
|---|---|---|---|---|
| `id` | `id` | `Int` | 主键；默认 autoincrement() | 内部自增主键 |
| `cycleId` | `cycle_id` | `Int` | 外键；与 `version` 联合唯一 | 所属绩效周期 |
| `version` | `version` | `Int` | 必须大于 0 | 周期内递增版本号；首次复制为 1 |
| `sourceConfigTemplateVersionId` | `source_config_template_version_id` | `Int?` | 外键；新周期必填，历史回填可空 | 来源配置模板版本；来源之后归档不影响快照 |
| `ratings` | `ratings` | `Json` | 必须为 JSON array | 固定 S/A/B/C 的名称、说明、连续区间和映射分快照；形状同配置模板版本 |
| `orgOwnerWeight` | `org_owner_weight` | `Decimal @db.Decimal(5, 2)` | 0～100；SCHEDULED 时必须大于 0 | 组织负责人基础权重 |
| `projectOwnerWeight` | `project_owner_weight` | `Decimal @db.Decimal(5, 2)` | 0～100；SCHEDULED 时必须大于 0 | 项目负责人基础权重 |
| `peerWeight` | `peer_weight` | `Decimal @db.Decimal(5, 2)` | 0～100；SCHEDULED 时必须大于 0 | 同部门同事基础权重 |
| `crossDeptWeight` | `cross_dept_weight` | `Decimal @db.Decimal(5, 2)` | 0～100；SCHEDULED 时必须大于 0 | 跨部门协作方基础权重 |
| `schedulePreset` | `schedule_preset` | `Json` | 必须为 JSON object | SELF / PEER / MANAGER 相对计划启动时间的开放和软提醒偏移快照 |
| `notificationRules` | `notification_rules` | `Json` | 必须为 JSON object | 任务开放与填写提醒的接收人、抄送人与提醒频率快照 |
| `createdByOpenId` | `created_by_open_id` | `String` | 非空 | 创建本快照版本的操作者 open_id |
| `createdAt` | `created_at` | `DateTime` | 默认 now() | 创建时间 |
| `updatedAt` | `updated_at` | `DateTime` | @updatedAt | DRAFT / SCHEDULED 阶段最近调整时间 |

### 关系

| Prisma 关系字段 | 类型 | 说明 |
|---|---|---|
| `cycle` | `PerfCycle` | `onDelete: Cascade`；周期是快照聚合根，正常业务只软删除周期 |
| `currentForCycle` | `PerfCycle?` | 被哪个周期选作 `currentConfigVersion`；唯一指针保证一个版本最多作为一个周期当前版本 |
| `sourceConfigVersion` | `PerfConfigTemplateVersion?` | `onDelete: Restrict`；保留准确来源版本，来源归档后历史仍可读取 |
| `formSnapshots` | `PerfCycleFormSnapshot[]` | `onDelete: Cascade`；表单快照不能脱离配置版本存在 |
| `stageResults` | `PerfStageResult[]` | 使用该配置快照计算的阶段结果；新配置版本重算时保留旧版本结果 |

### 索引、约束与生命周期

| 类型 | 字段 | 说明 |
|---|---|---|
| 唯一约束 | `cycleId, version` | 同一周期版本号不重复 |
| 唯一约束 | `id, cycleId` | 作为表单快照复合外键目标，阻止跨周期挂载 |
| 索引 | `sourceConfigTemplateVersionId` | 反查来源版本创建的周期快照 |
| CHECK | 统一计算契约 | 三类人工评估均从计分维度汇总；评级或分数由每个维度自己的 `scoringMethod` 决定 |
| CHECK | JSON 顶层类型 | `ratings` 为数组；`schedulePreset`、`notificationRules` 为对象 |
| CHECK | 四类关系权重 | DRAFT 可暂存 0～100；SCHEDULED 完整性检查要求四项为正且严格合计 100 |
| 触发器 | 来源版本 | 新周期复制或改换来源时必须仍为 PUBLISHED，并以共享行锁避免与归档并发穿透；ACTIVE 追加版本必须严格继承 current 版本的同一来源和连续版本号，来源后来归档仍可继续周期内独立版本链 |
| 触发器 | ACTIVE / ARCHIVED | ACTIVE 已有版本与 ARCHIVED 全部版本不可原地修改或删除；ACTIVE 只可为配置重算事务追加“current version + 1”的同来源新版本，再由同事务切换 current 指针 |

## perf_cycle_form_snapshots

Prisma model：`PerfCycleFormSnapshot`

用途：按周期配置版本和受控 D/M 职级前缀保存完整表单结构。快照包含 SELF、PEER、MANAGER 子表单、维度、核心标记、权重、维度字段及其稳定业务 key；模板后续发布新版本或归档不会改变周期内容。PROMOTION 不进入绩效周期快照。

### 字段说明

| Prisma 字段 | 数据库列 | 类型 | 约束/默认值 | 说明 |
|---|---|---|---|---|
| `id` | `id` | `Int` | 主键；默认 autoincrement() | 内部自增主键，也是参与人答卷选择表单的稳定身份 |
| `cycleConfigVersionId` | `cycle_config_version_id` | `Int` | 复合外键；与 `jobLevelPrefix` 联合唯一 | 所属周期配置版本 |
| `cycleId` | `cycle_id` | `Int` | 复合外键冗余字段 | 必须与所属配置版本的周期一致，用于数据库直接阻止跨周期挂载 |
| `jobLevelPrefix` | `job_level_prefix` | `PerfJobLevelPrefix` | D / M；配置版本内唯一 | 该快照准确适用的系统职级前缀 |
| `sourceFormTemplateVersionId` | `source_form_template_version_id` | `Int` | 外键；来源复制时必须为 PUBLISHED | 来源评估表单模板版本 |
| `content` | `content` | `Json` | 必须为 object，包含 `schemaVersion` 与 `subforms` array | 完整表单结构；保留来源层级 ID，并复制维度/字段稳定业务 key、计分方式与条件必填规则 |
| `createdAt` | `created_at` | `DateTime` | 默认 now() | 创建时间 |
| `updatedAt` | `updated_at` | `DateTime` | @updatedAt | DRAFT / SCHEDULED 阶段最近调整时间 |

- 旧 `PENDING` 因没有新版计划与配置快照而迁回 `DRAFT`；旧 `SELF_REVIEW / REVIEWING / AI_ANALYZING / CALIBRATING / CONFIRMING` 统一迁移为 `ACTIVE`。

- 可延迟约束触发器在事务提交时检查 SCHEDULED 周期的 D/M 快照、关系权重、参与人数和逐人前缀绑定。DRAFT 与 SCHEDULED 间调整不会生成提前可填写的任务。

- 归档只允许 `ACTIVE -> ARCHIVED`，且必须与唯一 `PerfCycleArchive` 快照同事务提交；`ARCHIVED` 后数据库拒绝周期 UPDATE / DELETE，不存在任何出站状态。

## perf_evaluation_submissions

统一人工评估答卷父记录。ACTIVE 周期发生表单结构变更时，旧 DRAFT/SUBMITTED 答卷转为 INVALIDATED 并完整保留，系统另建迁移兼容答案后的 DRAFT，避免覆盖历史证据。
