## 3. 配置体系

配置优先级：**环境变量 > YAML > 硬编码默认值**。加载器为 `backend/src/config/configuration.ts` 的 `loadAppConfig()`，采用"`config/app.yaml`（公共层）→ `dev.yaml` / `prod.yaml`（按 `NODE_ENV`）深度合并 → 环境变量覆盖"。

关键点：

- **`prisma.config.ts` 直接调用同一个 `loadAppConfig()`**，Prisma CLI 与 Nest 应用拿到完全一致的数据库连接串，改配置逻辑两端同时受影响。
- 可用 `APP_CONFIG_FILE` / `CONFIG_FILE` 临时指定 YAML。
- 环境变量校验见 `src/config/validation.schema.ts`（joi），覆盖 `NODE_ENV / PORT / DATABASE_URL / POSTGRES_URI / REDIS_URI / LARK_APP_ID / LARK_APP_SECRET / LARK_OAUTH_* / AUTH_JWT_*` 等。

配置树（`loadAppConfig()` 返回值）：

| 配置路径 | 环境变量 | 默认值 | 说明 |
|---|---|---|---|
| `app.port` | `PORT` | 3000 | 监听端口 |
| `database.url` | `DATABASE_URL`（兼容 `POSTGRES_URI`） | 本地串 | PG 连接串 |
| `redis.uri` | `REDIS_URI` | `redis://localhost:6379/0` | Redis 连接串 |
| `lark.appId` / `lark.appSecret` | `LARK_APP_ID` / `LARK_APP_SECRET` | — | 飞书自建应用凭证 |
| `lark.oauth.authorizeUrl / redirectUri / scope / webRedirectUrl` | `LARK_OAUTH_*` | 有默认 | OAuth 授权页、回调、登录成功后跳回前端的地址 |
| `auth.jwt.secret` / `auth.jwt.expiresIn` | `AUTH_JWT_SECRET` / `AUTH_JWT_EXPIRES_IN` | dev 默认 / `7d` | 应用会话 JWT |

> ⚠️ **已知风险**：`config/app.yaml` / `dev.yaml` 中当前硬编码了真实 `appId` / `appSecret`。生产环境必须用环境变量覆盖，且应尽快将真实密钥移出仓库（`prod.yaml` 已留空，做法正确）。

## 4. 已实现基建详解

### 4.1 认证与会话（`src/auth`）

登录采用**飞书 OAuth2 授权码模式**（网页外打开授权页，非 JSAPI 免登；工作台内免登为后续增强项，见 §8.9）。

流程：

1. 前端登录页调 `GET /auth/lark/authorize-url`，后端生成随机 `state` 存 Redis（TTL 600s）并返回飞书授权页 URL。
2. 用户在飞书授权后，飞书回调 `GET /auth/lark/callback?code&state`。
3. 后端 `redis.getdel` 一次性消费 state → SDK `authen.v1.accessToken.create` 换取 **user_access_token** → 组装用户信息（open_id/union_id/name/avatar 等）→ user_access_token 落 Redis → `jwtService.signAsync` 签发**应用会话 JWT**（`sub = open_id`）。
4. 后端 302 跳转前端 `/auth/callback?token=<jwt>&open_id&name&avatar`，前端写入 localStorage（`dingstock_token` / `dingstock_user`）后进入工作台。

Redis Key 约定：

| Key | 内容 | TTL |
|---|---|---|
| `auth:lark:state:{state}` | OAuth 防 CSRF state | 600s，一次性消费 |
| `auth:lark:user-token:{openId}` | `{accessToken, accessTokenExpiresAt, refreshToken}` | 取 `refresh_expires_in` |
| `auth:lark:jsapi-ticket:{openId}` | 用户级 jsapi_ticket | `expireIn - 300s` |

令牌刷新：`getUserAccessToken()` 提前 60s 判定过期，用 refresh_token 调 `authen.v1.refreshAccessToken.create` 刷新落库；无 refresh_token 或刷新失败抛 401 要求重新登录。

> **网页组件鉴权硬约束**：成员名片（`user_profile`）与人员搜索（`selector`）组件**仅支持用户身份鉴权**——必须用 user_access_token 换 jsapi_ticket，用 tenant_access_token 会报 `20442 jsapi-ticket not exist`。老会话没存用户令牌时需重新登录一次。

路由表：

| 方法 | 路径 | 说明 | JWT |
|---|---|---|---|
| GET | `/auth/lark/authorize-url` | 获取飞书授权页地址 | 否 |
| GET | `/auth/lark/callback` | OAuth 回调，302 跳回前端 | 否 |
| GET | `/auth/lark/jsapi-signature?url=` | 网页组件 JSAPI 签名（SHA-1，10 分钟一次性） | **是** |
| GET | `/auth/me` | 当前登录用户 | **是** |

`JwtAuthGuard`（`src/auth/jwt-auth.guard.ts`）解析 `Authorization: Bearer`，验证后把用户挂到 `request.user`。**后续所有业务模块的 controller 默认挂 `JwtAuthGuard`**。

### 4.2 组织架构同步（`src/contact`）

对应产品文档 §6.1。**全量同步**（每次拉全量后按主键 upsert），非增量：

1. **部门**：`contact.v3.department.childrenWithIterator`，从虚拟根 `"0"` + `fetch_child=true` 一次拉整棵树，upsert 到 `lark_departments`。ID 体系固定 `open_department_id` / `open_id`。
2. **员工**：逐部门调 `contact.v3.user.findByDepartmentWithIterator`（`user.list` 已废弃），按 open_id 去重后 upsert 到 `lark_users`。
3. **人事详情（增强层）**：`corehr.v2.employee.batchGet` 按 open_id 分批（≤100/批），约 70 个字段（职级、序列、直属上级、试用期、合同等），upsert 到 `lark_corehr_employees`。**未开通飞书人事或缺权限时自动降级**（记 `corehrError`，不影响整体成功）。

并发控制与状态：Redis `SET NX EX` 分布式锁 `contact:sync:lock`（TTL 1800s）防重复触发；同步状态写 `contact:sync:status` 供前端每 2s 轮询。

查询侧（`contact.service.ts`）：`listUsers` 支持按部门过滤，内存 BFS 展开"部门 + 全部子部门"ID 集合后用 PG 数组 `hasSome` 匹配；`include: corehr` 并批量反查直属上级姓名。

路由表（⚠️ 当前**未挂 JwtAuthGuard**，属技术债，见 §11）：

| 方法 | 路径 | 说明 |
|---|---|---|
| POST | `/contact/sync` | 触发全量同步（异步，立即返回） |
| GET | `/contact/sync/status` | 查询同步状态 |
| GET | `/contact/departments` | 已同步部门列表 |
| GET | `/contact/users?department_id=` | 已同步员工列表（含 corehr、直属上级姓名） |

### 4.3 共享基建（`src/shared`）

`SharedModule` 聚合并 re-export 全局 `ConfigModule`、`DatabaseModule`、`LarkModule`、`RedisModule`；**业务模块只需 `imports: [SharedModule]`**。

| 模块 | Token / Provider | 要点 |
|---|---|---|
| database | `PrismaService` | 继承 `PrismaClient`，`PrismaPg` 适配器直连；`onModuleInit` $connect / `onModuleDestroy` $disconnect |
| lark | `LARK_CLIENT`（Symbol）、`LarkService` | 业务代码**优先注入 `LarkService`**；`getJsapiTicket()` 强制走 user_access_token |
| redis | `REDIS_CLIENT`（Symbol） | `lazyConnect: true`；`REDIS_SHUTDOWN_HOOK` 退出时断开；业务勿 `new Redis()` |
| swagger | `setupSwagger(app)` | UI `/api-docs`，JSON `/api-docs-json`，已配 BearerAuth |

`main.ts` 启动流程：`enableShutdownHooks()` → `enableCors({origin:true, credentials:true})`（供 3001 前端跨域）→ `setupSwagger` → 按 `app.port` 监听。**当前未设全局 `ValidationPipe` 与路由前缀**（见 §11 技术债）。

调试脚本：`src/scripts/lark-playground.ts`（`pnpm lark:play`）——只加载 Config + Lark 模块的 SDK 试验场，改完即弃。

### 4.4 前端基建（`web/src`）

- **API 客户端** `lib/api.ts`：`apiFetch<T>(path, init)` 拼接 baseURL、自动附带 Bearer token、统一 JSON 解析与 `ApiError(status, message)`。所有接口调用必须经它。
- **Data Table 基建** `components/datatable/`：`DataTable`（渲染）、`DataTableToolbar`（搜索/全局搜索 + 筛选插槽）、`DataTableColumnFilter`（列筛选下拉）、`DataTablePagination`、`DataTableViewOptions`（列显隐）、`DataTableBulkActionBar`（批量操作条）、`DataTableExportButton`。**所有业务表格基于 `useReactTable`，列定义与页面组件分离**（`*-columns.tsx`）。
- **飞书网页组件** `components/shared/lark/` + `lib/lark-web-component.ts`：`UserAvatar`（头像+成员名片弹层，缺 open_id 自动退化为纯展示）、`LarkMemberSelector`（人员搜索，回调 `LarkSelectorOption`）、`useLarkComponentMount`（按需注入 h5-js-sdk、调 `/auth/lark/jsapi-signature` 完成一次性 `config`、跟随 next-themes 同步主题）。**人员头像/选人一律用这两个组件**，禁止裸用 `ui/avatar` 或自拼通讯录接口。
- **布局**：`app/(pages)/` 带侧边栏（`Sidebar` + `Header`），`app/(blank)/` 全屏（登录/回调）；`page.tsx` 只做薄壳，真实 UI 在 `src/views/`；导航唯一入口 `configs/navConfig.tsx`。

建模惯例（新表沿用）：

- 表名 `@@map` 为 snake_case 复数；字段 snake_case。

- 飞书侧枚举/嵌套对象以 `Int` / `JSONB` 原样存储；**业务域新表允许使用 Prisma enum**（语义封闭、由我方定义的状态字段）。

## 7. 绩效业务域数据建模方案

以下为待开发业务表的 Prisma 建模设计，对应产品文档 §8"核心数据对象"。原则：

1. **员工主数据不复制**：业务表统一以飞书 `open_id`（string）作为员工外键，展示层 join `lark_users`；周期启动时把部门/职级/Leader 写入**快照字段**，组织变动不影响历史（产品文档 §6.1.5、§6.6.4）。
2. **动态表单用 JSONB**：评估维度的表单 schema、维度评分明细、晋升评估三方填写项，均以 JSONB 存储，避免为每个周期的自定义配置改表。
3. **归档不可变**：`archived_at` 落定后业务代码禁止 UPDATE 结果表；任何调整必须经校准记录/操作日志产生新记录（产品文档 §3.5、§10.3）。
4. **状态字段用 Prisma enum**，与状态机（§8.2）一一对应。

### 7.1 表清单

| Model（建议） | 表名 | 对应产品对象 | 关键字段 |
|---|---|---|---|
| `PerfCycle` | `perf_cycles` | 8.4 绩效周期 | name、type(`SEMI_ANNUAL`，预留季/年)、start/end_date、status(周期状态机)、owner_open_id、时间窗口 JSONB（`windows`：各阶段起止/提醒，见产品 §9.3）、通知规则 JSONB |

### 7.2 关系与索引要点

- 一切业务表挂 `cycle_id`（直接或经 participant），周期删除仅允许 `DRAFT` 状态（产品 §5.9）。

## 8. 后端业务模块设计

### 8.1 模块规划（对应产品文档 §11.3 接口域）

```text
src/
├── auth/            ✅ 已有
├── contact/         ✅ 已有（Organization 接口域）
├── cycle/           周期 + 评估规则 + 评估维度 + 时间窗口（Cycle/Dimension 域）
├── participant/     考核人员管理（Participant 域）
├── review/          评审员指派/推荐、自评、360°、上级评估（Task/SelfReview/Review 域）
├── calibration/     校准 + 最终结果 + 等级分布（Calibration/Result 域）
├── appeal/          申诉 + 面谈（Appeal/Interview 域）
├── ai-report/       AI 分析异步任务（AIReport 域）
├── dashboard/       看板统计 + 指标（Dashboard 域）
├── report/          报表导出任务（异步生成 xlsx）
├── notification/    飞书消息推送 + 催办调度（Notification 域）
└── audit/           操作日志（AuditLog 域，提供拦截器/装饰器给各模块）
```

每个模块 `imports: [SharedModule]`，controller 默认挂 `JwtAuthGuard` + 角色守卫（§8.4）。

### 8.2 状态机实现

产品文档 §5.9 定义了周期级状态机。落地拆成两层：

**周期状态**（`PerfCycle.status`）：`DRAFT → PENDING → SELF_REVIEW → REVIEWING → AI_ANALYZING → CALIBRATING → CONFIRMING → ARCHIVED`。由 HR 操作（启动/关闭）与时间窗口调度（`@nestjs/schedule` cron 扫描 `windows`）共同驱动。

**参与者状态**（`PerfParticipant.status`）：跟随个体进度 `PENDING_SELF_REVIEW → SELF_SUBMITTED → REVIEWED → AI_DONE → CALIBRATED → RESULT_PUSHED → CONFIRMED / APPEALING → RE_CONFIRMING → ARCHIVED`，兼容退回（`RETURNED`）分支。周期整体状态按参与者聚合推进（如全部归档 → 周期 ARCHIVED）。

实现方式：在各模块 service 内实现**显式流转函数**（`transition(participant, action)`），集中定义"当前状态 × 动作 → 下一状态"映射表，非法流转抛 `ConflictException`；所有流转写 `AuditLog`。不引入重量级状态机库，映射表即文档。

异常回退规则按产品文档 §5.8 实现：退回自评（`RETURNED`）、更换评审员（assignment 置 `REPLACED` 新建记录）、HR 开放退回已提交评审（记录原因）、延长窗口（改 `windows` JSONB + 日志）。

### 8.4 权限（RBAC）实现

对应产品文档 §3。三层判定：**角色（RoleGrant + 派生角色） × 组织范围（org_scope / leader 快照） × 任务关系（assignment / participant）**。

落地：

1. `RolesGuard` + `@Roles('HR')` 装饰器控制接口级权限；
2. service 层做**数据范围过滤**：员工只查 `employee_open_id = me`；评审员只查 `reviewer_open_id = me` 的 assignment；Leader 按 `leader_open_id 快照 = me`；HR 按 `org_scope` 过滤部门子树（复用 contact 的子树展开逻辑，建议下沉为共享工具并加 Redis 缓存）；
3. 字段级敏感控制（薪酬、晋升结论、AI 报告可见性）在 DTO 序列化层按角色裁剪，禁止把整行实体直接返回。
4. 员工在结果推送前不可见任何评分（产品 §3.2）：结果类接口一律以 `RESULT_PUSHED` 之后的状态为可见前提。

### 8.5 通知与催办调度

- 发送通道：飞书机器人私聊消息/卡片（`LarkService` → `im.v1.message.create`），卡片模板集中管理（评审启动/自评提醒/任务通知/考核 Leader 职责转入转出/催办/结果推送/申诉/面谈/归档，产品 §9.10）。
- 调度：引入 `@nestjs/schedule`；cron 扫描各周期 `windows` 与参与者状态，按催办规则（截止前 N 天/当天/逾期每日，产品 §9.9）生成发送任务。
- 可靠性：每次发送落 `perf_notifications`；失败进重试（指数退避，`retry_count` 上限后置 `FAILED` 支持手动补发，产品 §5.8）。
- 幂等：以 `receiver + template + 业务键 + 日期` 做去重键（Redis SETNX），避免重复打扰。

### 8.6 AI 分析异步任务

对应产品文档 §5.5、§10.6：

1. 触发：参与者进入"评审数据齐备"（上级评估提交）后自动创建任务；也支持手动重跑。
2. 执行：组装上下文（自评/总结/晋升自述/360°/上级评分评语/历史绩效）→ 调用 AI 服务生成六段输出（摘要/亮点/待改进/晋升证据/一致性提示/风险提示）→ 落 `perf_ai_reports`（含 `inputs_digest` 追溯）。
3. 队列：首期用"DB 任务表 + cron 轮询 + Redis 锁"轻量实现即可（与报表导出共用模式），量级不需要引入 BullMQ；若后续吞吐增长再升级。
4. 约束：失败重试、AI 开关（系统参数）关闭时整段跳过且流程直达校准、输出仅 Leader/HR 可见（员工可见性按配置）、人工结论优先。

### 8.7 报表导出

异步任务模型（同上共用）：`POST /reports/export` 建任务 → worker 用 SQL 聚合生成 xlsx（后端生成，前端 `xlsx` 依赖仅用于兜底小导出）→ 产物落盘并记录下载地址 → 前端任务列表轮询。导出动作必须写 `AuditLog`（产品 §6.8.2）。

### 8.8 指标管理（产品 §6.2）

首期不建独立"指标引擎"：内置指标（自评提交率/评审完成率/校准完成率/确认率/申诉率/面谈完成率/晋升覆盖率）以 dashboard 模块内的 SQL 聚合函数 + Redis 缓存（TTL 60s）实现，指标口径集中写在一个 `metrics.ts` 供看板与报表复用。指标库 CRUD 留到四期。

### 8.9 飞书集成增强（二期起）

- **工作台内免登**：检测飞书容器（User-Agent / `window.h5sdk`）时走 JSAPI `tt.requestAccess` 静默换 code，复用现有 `loginWithAuthorizationCode`。
- **事件订阅**：`contact.user.updated / department.updated` 等事件走长连接或回调地址，触发增量 upsert（替代当前"手动全量"）；同时新增每日定时全量兜底（cron 02:00）。
- **OKR 读取**：`okr.v1` 接口按员工拉取周期 OKR，自评页引用（产品 §6.4.3，可选能力）。
