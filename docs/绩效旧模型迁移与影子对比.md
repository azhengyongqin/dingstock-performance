# 绩效旧模型迁移与影子对比运行手册

> 历史手册：本页记录 Ticket 20 扩展期曾执行的迁移流程。Ticket 21 已完成 contract，仓库不再提供 `migration:legacy` runner，也不能回退到旧读写；`perf_legacy_migration_runs/items` 仅保留上线审计。当前运维请使用《绩效新模型切换与运维》。

本手册对应 Ticket 20 的历史操作。迁移曾采用“扩展 → 验证 → 后续切换”方式：runner 只向版本化模板、周期快照、统一提交、关系化评估项、阶段结果、结果版本和迁移账本加法写入；当时不会设置 `perf_cycles.current_config_version_id`。

## 迁移对象与稳定业务键

| 旧对象 | 新对象 | 来源业务键 |
|---|---|---|
| `PerfTemplate` / 周期内实际维度 | 不可变 D/M `PerfFormTemplateVersion`、`PerfConfigTemplateVersion` | `template:{id}`；无模板时 `cycle:{id}` |
| `PerfCycle` 的规则、窗口、维度 | `PerfCycleConfigVersion` + D/M `PerfCycleFormSnapshot` | `perf_cycles:{id}` |
| `PerfSelfReview` | `PerfEvaluationSubmission(SELF)` + item results | `perf_self_reviews:{id}` |
| `PerfReview` | `PerfEvaluationSubmission(PEER)` + item results | `perf_reviews:{id}` |
| `PerfManagerReview` | `PerfEvaluationSubmission(MANAGER)` + item results | `perf_manager_reviews:{id}` |
| 旧 PEER/MANAGER JSON | `PerfStageResult` + dimension/relation aggregates | `participant:{id}/stage:{stage}` |
| `PerfResult` | `PerfResultVersion` | `perf_results:{id}` |

`perf_legacy_migration_items` 对 `(source_type, source_business_key)` 建唯一约束，并保存规范化来源 JSON 的 SHA-256。相同来源重跑复用目标记录；来源在首次成功后变化会以 `SOURCE_CHANGED_AFTER_MIGRATION` 阻断，不会静默覆盖新模型。

## 1. 单周期 dry-run 演练

```bash
cd backend
pnpm migration:legacy -- plan --run-key=t20-cycle-123-plan --cycle-id=123
```

成功标志：命令退出码为 0，JSON 中 `dryRun=true`；`readiness.blockers` 必含 `DRY_RUN_ONLY`。检查 `validationReport.issues` 是否包含职级前缀、状态歧义、未知维度、重复维度、分数越界、缺少评审指派或旧自评等级。

## 2. 单周期 apply 与失败重跑

```bash
pnpm migration:legacy -- apply --run-key=t20-cycle-123-apply --cycle-id=123
pnpm migration:legacy -- report --run-key=t20-cycle-123-apply
```

成功标志：批次为 `COMPLETED`；每个来源项为 `MIGRATED` 或有可定位的 `FAILED.detail`。`run-key` 固化一次执行及报告，`COMPLETED` 后再次调用只返回已存报告。修复旧主数据/JSON 后应创建**新的 run-key**再次 apply；checksum 未变化的成功项会通过全局来源账本复用，不重复写。异常中断形成的 `FAILED` 批次才允许用同一 run-key 条件领取重跑，且并发只能有一个执行器成功领取。

注意：旧 `PerfSelfReview` 没有固定评级列。runner 只接受 `summary` 或 `okrContent` 中可验证的 `selfLevel/selfRating/level/rating = S/A/B/C`。缺失时记录 `MISSING_LEGACY_SELF_LEVEL`，不得凭空创建自评答案历史。

## 3. 全量迁移与影子报告

```bash
pnpm migration:legacy -- apply --run-key=t20-full-20260716
pnpm migration:legacy -- report --run-key=t20-full-20260716
```

影子报告 `shadowReport.rows` 每行包含：参与者业务键、人数定位字段、旧人工上级等级、新计算等级、是否不同、差异原因和处置状态。确认差异符合新规则后，可针对明确业务键重跑并标记已接受：

```bash
pnpm migration:legacy -- apply \
  --run-key=t20-full-20260716-shadow-accepted-1 \
  --accept-shadow-key=cycle:123/employee:ou_xxx
```

不得用批量“全部接受”掩盖差异；每个业务键都应有产品/HR 的可追溯处置依据。

## 4. readiness 硬门槛

```bash
pnpm migration:legacy -- assert-ready --run-key=t20-full-20260716
```

成功标志：命令退出码为 0 且返回 `ready=true`。门槛全部为零容忍：

- 旧/新周期、正式有效提交、关系化 item、维度/关系结果和结果版本计数一致；
- 没有缺失业务键或 D/M 职级前缀；
- 没有无效维度 JSON、未闭合状态或迁移失败项；
- 所有新旧上级等级差异均已逐人处置。

Ticket 21 在此命令成功前不得设置新读路径或停止旧写入。

## 5. 补偿回滚演练

```bash
pnpm migration:legacy -- rollback --run-key=t20-cycle-123-apply
pnpm migration:legacy -- report --run-key=t20-cycle-123-apply
```

成功标志：run 与 items 均标记 `ROLLED_BACK`，readiness 变为 false 并包含 `RUN_ROLLED_BACK`。扩展期回滚采用 `DETACH_ONLY`：由于从未切 `current_config_version_id`，旧读取天然继续工作；新不可变模板、提交和结果产物不物理删除，保留作为审计证据，也避免破坏可能已被后续历史引用的数据。回滚批次不可直接复用，修复后创建新的 run-key 重新演练。

## 6. PostgreSQL 演练建议

1. 在生产快照恢复出的隔离数据库执行全部 Prisma migration。
2. 记录三组基线 SQL：旧表 count、旧业务键去重 count、状态分布。
3. 依次执行 plan、单周期 apply、同 key 重跑、全量 apply、assert-ready。
4. 对迁移账本唯一约束做并发重跑测试；确认没有重复统一提交或结果版本。
5. 执行 rollback；用旧接口读取同一周期，确认返回内容与演练前一致。

成功标志：迁移 SQL 全部应用、同 key 两次 apply 目标计数不变、报告异常均可按来源业务键定位、rollback 后旧读取仍可用。
