# 红线由 HR 或管理员正式认定

重大事故、严重违规或红线问题不作为上级评估表中的普通复选框。Leader 可以针对参与者上报疑似红线及说明，但只有 HR 或管理员可以作出正式认定。认定必须关联参与者与绩效周期，并保存红线类型、事实说明、证据附件、认定人和认定时间；只有已确认且仍有效的红线记录进入上级评估计算，将其阶段等级、校准前等级和最终等级强制为 C。绩效校准或申诉处理不能直接把有效红线下的等级调高；如调查推翻认定，必须先由 HR 或管理员撤销红线并填写原因，再重新计算上级评估阶段结果并进行后续校准。认定的新增、修正和撤销均需保留完整审计链。

**Considered Options**：允许 Leader 在提交上级评估时直接勾选红线，交互更快，但一次表单操作就能绕过全部评分算法并强制定级，且难以区分线索上报与正式组织认定。把线索提交和 HR/管理员认定分开，可以保留管理者的事实输入，同时为最严厉的等级约束提供权限与证据边界。

**Consequences**：数据库需要独立的参与者红线业务记录或等价的可审计模型，不能只在评估提交 JSON 中保存布尔值；上级评估计算服务与最终结果服务都必须读取有效认定，评估表只展示红线状态与计算原因。权限、附件、撤销后重算、校准门控和审计查询都需要纳入实现与验收。
