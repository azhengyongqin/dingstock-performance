# 校准采用乐观并发控制并拒绝静默覆盖

同一参与者可能同时被直属 Leader、HR 或 Admin 打开和校准。校准提交必须携带操作者进入页面时看到的最新校准结果标识（首次校准则明确表示不存在前序结果）。服务端在同一事务内检查该标识是否仍为最新；若其他人已经成功追加了更新的校准结果，则拒绝当前提交，要求操作者刷新并比较差异。刷新后仍可显式重新校准，且继续追加新记录。当前生效校准结果取最新一次成功提交的记录，完整历史永久可见。

**Considered Options**：直接采用“最后写入者生效”实现简单，但会让后提交者在不知情时覆盖他人的正式绩效决定，尤其直属 Leader、HR 和 Admin 权限重叠时审计风险较高。悲观锁会长时间占用资源且无法覆盖用户在页面上停留的时间。乐观版本检查能以较低成本暴露冲突，并让操作者基于最新信息主动决定是否重新校准。

**Consequences**：校准接口需要接收预期的最新校准标识，并用事务或等价原子条件完成校验和写入；冲突返回明确的业务错误及最新结果摘要。前端必须提示“结果已被他人更新”，展示原页面版本与当前版本差异，并禁止自动重试。批量校准也必须逐人检查版本，冲突人员失败不能被批量操作静默覆盖。
