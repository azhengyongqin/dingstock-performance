# 评估阶段按结果模式生成阶段等级

为落实公司现行的绩效等级计算方案，我们决定用本 ADR 取代 ADR-0001：每个评估阶段采用直接定级、加权评级或加权评分中的一种阶段结果模式。直接定级由评估人选择阶段等级，其中员工自评等级为提交必填项，但只表达员工自我判断；加权评级将各维度等级映射为分值后按权重计算；加权评分直接按维度分数与权重计算。两种加权模式统一生成 0～100 的计算分：评级模式保存原始评级及当期映射分，评分模式保存原始分数且计算分等于原始分数。两种加权模式都要求适用维度权重合计为 100%，并在综合得分映射出初始等级后应用核心维度与红线约束，得到阶段等级。因此，维度权重从展示信息变为阶段计算规则的一部分。上级评估的阶段等级必须由可复算的规则生成，Leader 不可在评估表单中手工改写；需要人为调整时统一进入绩效校准并留痕。

阶段模式能力固定为：员工自评与 AI 评估只能直接定级；360°评估可以选择加权评级或加权评分，默认加权评级；上级评估可以选择加权评级或加权评分，默认加权评分。360°与上级评估不允许直接定级，避免绕过维度填写和约束计算。模式配置属于配置模板并随绩效周期固化。

**Considered Options**：继续由 Leader 脱离维度结果手工选择初始等级，会使公司已明确的加权算法、权重和约束规则失去业务作用；只把计算结果作为可覆盖建议，又会产生两个互相竞争的阶段结果口径，并绕开既有校准审计。各阶段等级如何形成参与者最终等级，作为独立决策另行确定。

**Consequences**：模板与周期配置必须能表达阶段结果模式、等级映射分和约束规则；360°评估与上级评估提交时必须保存原始填写值、统一计算分、所用规则快照与阶段结果。现有上级评估手工选择 `initialLevel`、权重仅展示的实现需要重构；前端只展示计算预览与约束原因，不提供手工等级选择器。进行中周期修改计算配置时按 ADR-0026 生成新配置版本并统一重算，不能继续沿用旧计算结果。
