## 1. 设计原则

1. **角色导向**：员工 / 评审员 / Leader / HR / 管理员共用一套导航框架，按角色看到不同分组与数据范围（产品文档 §3）。同一页面对不同角色只做数据过滤，不做两套 UI。
2. **流程可视**：绩效周期是六阶段流水（评审启动 → 员工自评 → 评审打分 → AI 分析 → 确认申诉 → 面谈闭环）。所有周期相关页面用**步骤条 / 阶段徽标**让用户随时知道"现在在哪一步、我要做什么"。
3. **参考信息左置，填写动作右置**：一切评估填写场景（自评、360°、上级评估）沿用飞书绩效的经典左右分栏——左侧只读参考材料，右侧评分表单（飞书绩效参考 `image 1/5/7.png`）。
4. **条件展示**：晋升评估是绩效周期内的可配置维度，不是独立流程。仅当"周期启用 + 员工在范围 + 角色被授权"三条件齐备时展示晋升区块（产品文档 §6.4.5），且以明显的分区标题与说明区隔。
5. **闭环可追溯**：结果确认、申诉、面谈、调整全部留痕并在 UI 呈现（时间线 / 状态标签 / 调整前后对比），传达"过程可信"。
6. **飞书原生感**：人员、部门等实体一律用飞书网页组件呈现（头像名片、人员搜索），通知触达走飞书卡片，减少"另一个系统"的割裂感。

## 2. 设计系统基础

### 2.1 布局框架

- **带侧边栏布局** `app/(pages)/`：左侧固定侧边栏（Logo + 分组导航，`Sidebar`）+ 顶部 Header（面包屑/搜索 `CommandMenu`、通知 `NotificationDropdown`、主题切换 `ModeToggle`、头像菜单 `ProfileDropdown`）+ 内容区。内容区统一 `flex flex-col gap-6`，页面顶部为 `PageHeader`（标题 + 描述 + 右侧动作区）。
- **全屏布局** `app/(blank)/`：登录、OAuth 回调等无导航页面。
- 内容区栅格：统计卡行用 `grid sm:grid-cols-2 lg:grid-cols-4`；主次分栏用 `lg:grid-cols-3`（主 2 副 1）或 `lg:grid-cols-[280px_1fr]`（导航型左栏）。

### 2.2 主题与色彩

- 主题系统来自模板：next-themes 明暗切换 + `themeConfig` / `theme-presets` 预设色 + `settingsContext` 持久化。**所有颜色必须用语义 token**（`primary` / `muted` / `destructive` / `border` 等），禁止硬编码色值，保证暗色模式与换肤可用。
- 飞书网页组件跟随主题：`useLarkComponentMount` 已随 next-themes 自动 `webComponent.update({ theme })`，新增飞书组件必须走该 hook。
- **状态语义色**（Badge 统一用法 `bg-{color}-500/10 text-{color}-600 dark:text-{color}-400`）：

| 语义 | 颜色 | 用例 |
|---|---|---|
| 进行中 / 信息 | blue | 周期进行中、面谈中 |
| 待处理 / 警示 | yellow | 待确认、待校准、待处理申诉 |
| 逾期 / 失败 / 紧急 | red | 逾期任务、同步失败、紧急徽标 |
| 完成 / 成功 | green（或 `primary/10`） | 已提交、已解决、已归档 |
| 中性 / 草稿 | outline 变体 | 草稿、未开始、角色标签 |

- **绩效等级色**：等级徽标 D/C/B/A/S 当前统一 `primary` 色系（`bg-primary/10 text-primary`），大号等级展示用 `size-20 rounded-2xl text-4xl` 的等级方块（结果确认页范式）。若后续需要区分档位色（如 S 金、D 红），在 columns 层集中定义，禁止散落各页。

### 2.3 组件规范

- 组件库：shadcn/ui（base-vega 风格）位于 `components/ui/`；业务共享组件位于 `components/shared/`。缺组件先查模板，再考虑用 shadcn CLI 添加，不手写平替。
- **表格（强制）**：一律 `useReactTable` + `components/datatable/` 基建；列定义拆分到 `*-columns.tsx`。变体映射见 §5.1。
- **人员（强制）**：头像/姓名 → `UserAvatar`（点击弹飞书成员名片，缺 open_id 自动退化纯展示）；选人/搜人 → `LarkMemberSelector`（飞书搜索组件，固定搜"人"）。
- 图表：Recharts 经 `ui/chart.tsx`（`ChartContainer` + `ChartTooltip`）封装，配色用 `var(--primary)` 及其 `color-mix` 衰减档（40%/60% 等）构成同色系序列。
- 反馈：操作结果用 `sonner` toast（成功 `toast.success`、异步开始 `toast.info`、失败 `toast.error`）；破坏性操作先 `Dialog` 确认。

### 2.4 通用状态设计（每个接数据的页面必须齐备）

| 状态 | 规范 | 范例 |
|---|---|---|
| 加载 | `Skeleton` 占位，形状贴近最终内容（列表 5 行 `h-12`，树 3 条递减宽度） | 组织架构页 |
| 错误 | 虚线边框卡片 + 错误说明 + 「重试」按钮；文案给出可行动指引（如"请确认后端服务已启动"） | 组织架构页 |
| 空态 | 虚线边框卡片 + 主文案 + 引导动作（如"点击右上角同步组织架构"） | 组织架构页 |
| 表格空 | `DataTable` 的 `emptyText`，文案区分"没有数据"与"没有匹配结果" | 各列表页 |

## 4. 核心用户旅程（页面串联）

以一个完整周期为例（产品文档 §5.1 泳道）：

```text
HR：/cycles/new/edit（9 步向导）→ 启动 ──飞书卡片──▶ 全员
员工：卡片/工作台 → /self-review 填写 → 提交
Leader：/team-review → 指定评审员（/cycles/:id/reviewers）
评审员：卡片 → /review-tasks → /review-tasks/[id] 打分提交
Leader：/team-review → /team-review/[pid]/evaluate 上级评估 → 查看 AI 报告
HR/Leader：/calibrations 校准 → 推送结果 ──飞书卡片──▶ 员工
员工：/results 确认 ─┬─ 无异议 → 归档 → /profile/performance 沉淀
                    └─ 申诉 → HR/Leader /appeals/[id] 面谈 → 再次确认 → 归档
```

设计要求：每一步完成后的"下一步去哪"必须由系统给出（飞书卡片按钮直达 + 工作台待办），不依赖用户记路径。

## 5. 通用交互模式

### 5.1 Data Table 变体映射（强制，来自 CLAUDE.md）

| 业务场景 | 变体 | 已应用页面 |
|---|---|---|
| 带状态/类别列、需按列过滤 | filters（`DataTableToolbar` 搜索 + `DataTableColumnFilter`） | 周期列表、申诉处理 |
| 完成度/进度语义 | progress（进度条列） | 团队看板 |
| 需要批量操作 | 行选择 + `DataTableBulkActionBar` | 绩效校准 |
| 列很多、用户自选列 | column-visibility（`DataTableViewOptions`） | 周期详情-考核人员 |
| 日志/流水大数据量 | page-size-selector + `DataTableExportButton` | 操作日志 |
| 简单小表 | basic（可加分页） | 报表导出任务、评审任务、周期详情各配置表 |
| 企业级完整列表 | users-list 全套（全局搜索 + 筛选 + 列显隐 + 分页） | 组织架构成员表 |

新页面先在此表找最贴近变体；没有再去模板 `datatables/` 挑（expandable-rows、pinnable 等），并把新映射补回 CLAUDE.md 与本表。

### 5.2 评估填写布局（左参考 + 右填写）

来源：飞书绩效参考 `image 1.png`（自评）、`image 7.png`（360°）、`image 5/12/4.png`（上级评估）。

- 桌面 ≥ lg：`grid lg:grid-cols-[minmax(320px,2fr)_3fr]`。左栏参考信息可独立滚动（`ScrollArea`），右栏为评分表单；左栏内容用 `Accordion`/`Tabs` 分节（自评总结 / OKR / 360° 反馈 / 历史绩效 / 任职信息 / HR 备注），默认展开最相关一节。
- 移动 < lg：左栏折叠为顶部 `Tabs` 或抽屉入口，填写区优先。
- 右栏评分表单：每个维度一个分区卡（维度名 + 权重徽标 + 填写说明 tooltip + 评分控件 + 评语 `Textarea`）。评分控件按维度 `scoring_method`：等级 → `ToggleGroup`（D/C/B/A/S 单选），分值 → `Input[type=number]` 或 `Rating`，结论型 → `RadioGroup`，纯文本 → 仅 `Textarea`。
- 必填校验：提交时定位到首个未通过分区并滚动聚焦；低分/高分需事实依据的规则以内联错误提示呈现（产品文档 §6.3.2）。

### 5.3 底部固定操作栏（长表单）

自评、360°、上级评估等长表单页固定底栏（`fixed inset-x-0 bottom-0 border-t bg-card`，内容 `pb-20` 预留）：左侧为流程提示文案（"提交后将进入 360° 评估环节…"），右侧「保存草稿」（outline）+「提交」（primary）。草稿自动保存（防抖 ≥2s）+ 手动保存双保险，保存成功以轻量 toast 或"已保存 HH:mm"文案反馈。

### 5.4 步骤条

- **周期阶段步骤条**（工作台）：圆形序号 + 连接线，完成态实心 `primary` + ✓，当前态描边 `primary`，未开始灰。
- **配置向导步骤条**（周期编辑）：左侧纵向步骤导航（9 步，标题 + 一行描述），可点击跳转；最后一步为"启动前检查"清单。

### 5.5 抽屉与弹窗分工

- **Sheet（抽屉，右侧）**：查看型下钻——校准页员工详情（评分构成 / 360° 摘要 / AI 摘要 / 历史绩效）、申诉详情、360° 反馈明细。保持列表上下文不丢失。
- **Dialog（弹窗）**：短决策型输入——校准调整等级（新等级 + 必填调整原因）、退回自评（原因）、更换评审员、发起导出。
- 破坏性/不可逆操作（启动周期、推送结果、归档）：Dialog 内复述影响范围（"将向 126 人推送结果通知"）再确认。

### 5.6 人员呈现

- 表格人员列：`UserAvatar`（size sm）+ 姓名；hover/点击出飞书名片。
- 选人：`LarkMemberSelector`；批量圈人（考核人员）用"部门树勾选 + 名单导入 + 选人组件"组合。
- 前置条件提示：网页组件需用户级鉴权，登录态失效或老会话未存用户令牌时组件降级为普通展示，并引导重新登录（研发文档 §4.1）。

### 5.7 通知触达（飞书卡片）

所有卡片遵循同一骨架：标题（事件）+ 关键信息（周期 / 截止时间 / 对象人数）+ 主按钮直达对应页面（如「去自评」→ `/self-review`）。催办卡片带剩余天数与逾期红色强调。页面内以工作台待办与 Header `NotificationDropdown` 呼应，三处信息一致。

## 6. 逐页设计说明

> 每页给出：目标与角色 / 布局与组件 / 关键交互 / 现状。「骨架 ✅」= 页面结构与交互框架已实现、数据为 mock；「待新建」= 尚无页面，按本节方案开发。

### 6.1 登录与回调（`/auth/login`、`/auth/callback`）— 骨架 ✅（已接后端）

全屏布局居中卡片。登录页单一主按钮「使用飞书登录」→ 跳转飞书授权页；回调页两态：加载（spinner + "正在完成登录…"）/ 失败（`CircleAlert` + 原因 + 「返回登录」）。成功后 `router.replace('/workbench')` 防回退。

### 6.2 工作台（`/workbench`）— 骨架 ✅

全员首页。结构：周期进度卡（当前周期名 + 时间窗 + 参评人数 + 状态徽标 + **六阶段步骤条**）→ 双栏：「我的待办」（角色徽标 + 任务标题 + 截止时间 + 紧急徽标 + 「去处理」直达）+「快捷入口」（2×3 图标宫格）。接真数据后：待办按当前用户角色聚合生成，无待办展示空态插画 + "本阶段无需处理"。

### 6.3 周期列表（`/cycles`）— 骨架 ✅

HR 管理入口。`PageHeader` 右侧「新建周期」。filters 变体表格：周期名搜索 + 状态列筛选 + 分页；列：名称（链接进详情）/ 类型徽标 / 状态徽标 / 参评人数 / 时间窗口 / 操作。操作列按状态呈现：草稿 → 编辑/删除；进行中 → 查看/关闭；已归档 → 查看/导出。

### 6.4 周期详情（`/cycles/[id]`）— 骨架 ✅

Tabs 组织：概览（统计卡 ×4：参评人数/自评完成率/评审完成率/当前阶段）/ 考核人员（column-visibility + 分页变体，按部门汇总）/ 评估维度（basic）/ 时间窗口（basic，阶段状态徽标）/ 通知规则（basic）。待补：概览 Tab 增加阶段步骤条与"启动前检查"结果回显；考核人员支持下钻到人（复用组织架构成员表模式）；新增「评审员指派」Tab（见 §6.7）。

### 6.5 周期创建/编辑（`/cycles/new/edit`、`/cycles/[id]/edit`）— 骨架 ✅

9 步向导（产品文档 §7.4）：基础信息 → 考核人员 → 评估规则 → 评估维度 → 晋升评估维度 → 评估角色 → 时间窗口 → 通知规则 → 启动前检查。左侧纵向步骤导航（可点击跳转），右侧当前步表单，底部「上一步/下一步」，末步「启动周期」。
待细化的各步表单要点：

- 考核人员：左部门树（复用组织架构页树组件）+ 右已选清单表格；支持按职级/岗位筛选、名单导入、逐人标记"参与晋升评估"。
- 评估规则：评级表（评级 / 名称 / 分值区间 / 备注，可增删行）+ 评语必填评级多选 + 预览弹窗。
- 评估维度：维度卡片列表（名称 / 类型 / 评分方式 / 权重 / 适用角色 / 必填 / 填写说明与示例），权重合计实时校验 = 100%。
- 晋升评估维度：顶部说明条（**"晋升评估随绩效周期一起启动，不会创建独立晋升流程"**，产品文档 §7.6 要求明示）+ 启用开关 + 适用范围圈选 + 员工/评审员/Leader 三方填写项配置 + 结论选项 + 员工结果页可见开关。
- 启动前检查：清单逐项校验（人员已圈定 / 权重 100% / 窗口无冲突 / 角色已配置 / 通知已配置），未通过项红色并可点击跳回对应步骤；全部通过才允许「启动周期」（二次确认弹窗复述影响）。

### 6.6 组织架构（`/settings/organization`）— ✅ 已接真实后端（全站范式页）

左 300px 部门树（懒展开、选中高亮、成员数徽标、再次点击取消筛选）+ 右成员表（users-list 全套：姓名/邮箱/工号全局搜索 + `LarkMemberSelector` 飞书搜人定位 + 在职状态筛选 + 列显隐（CoreHR 次要列默认隐藏）+ 分页）。`PageHeader` 右侧「同步组织架构」：触发后按钮 spinner + 2s 轮询状态 + 完成 toast 并刷新。加载/错误/空三态齐备——**新页面接数据一律照此范式**。

### 6.7 评审人推荐与指定（周期详情 Tab 或 `/cycles/[id]/reviewers`）— 待新建

- 暗色模式：全站必须通过暗色走查（语义 token 已保证大部分），重点检查图表 `color-mix` 序列与状态色 `dark:` 变体。

## 9. 文案规范

- 语言：全站简体中文；日期 `YYYY-MM-DD`，区间 `A ~ B`；人数 `N 人`；百分比整数。
- 状态词全站统一：未开始 / 草稿中 / 进行中 / 已提交 / 已退回 / 待校准 / 已校准 / 已调整 / 待确认 / 已确认 / 待处理 / 面谈中 / 已解决 / 已归档 / 生成中 / 已完成 / 失败。新增状态先补入本表再使用。
- 引导文案给出下一步动作（"可点击右上角「同步组织架构」…"），错误文案给出原因与自救方式，避免裸抛"请求失败"。
- 晋升相关文案必须体现"随绩效周期进行、非独立流程"的定位，避免"晋升申请""晋升审批"等误导词（本期不做任命与薪酬落地，产品文档 §2.5）。
