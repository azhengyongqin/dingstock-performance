# 绩效新模型切换与运维

本手册对应 Ticket 21 的最终 contract 状态。系统只使用版本化配置/表单、周期快照、统一人工提交、阶段结果、校准决定和不可变结果版本；旧模板、旧周期配置列、分离人工评估表及其枚举已经删除。

## 1. 切换门禁

contract migration `20260716170000_contract_legacy_performance_model` 会在任何删表前执行硬检查：

1. 必须存在最近一次 `dry_run=false`、`status=COMPLETED` 且 `readiness_report.ready=true` 的全量迁移批次。
2. 迁移账本必须能为所有历史周期定位 `CYCLE_CONFIGURATION` 目标版本。
3. 回填后任何周期缺少 `current_config_version_id` 都会让 migration 整体失败。
4. 全部条件通过后才删除旧表、旧列和旧 enum，并写入 `system_configs.performance.model.cutover`：`phase=CONTRACTED`、`readPath=VERSIONED`、`writePath=UNIFIED_SUBMISSION`、`rollbackEnabled=false`。

成功标志：Prisma migration 成功提交，`performance.system_configs` 中切换配置为最终态，应用构建和 Ticket 21 PostgreSQL 旅程测试通过。

## 2. 上线步骤

1. 备份数据库并记录当前 migration 版本、迁移批次 ID 与 readiness 报告。
2. 在生产快照恢复出的隔离数据库先执行全部 migration 和 `pnpm test:postgres`。
3. 确认切换配置、失败迁移项、影子差异、超时任务和通知失败指标。
4. 生产执行 migration；成功后部署同一提交的 backend 和 web，不允许混用扩展期应用版本。
5. 用 HR/Admin 身份访问 `GET /performance-cutover/status`，保存响应到变更单。
6. 抽查一名 D、一名 M 员工的 SELF / PEER / MANAGER 提交、校准、结果确认与历史查询。

成功标志：状态接口 `ready=true`；旧 `/templates`、旧自评/上级评估接口返回 404；新模板、周期、看板、通知和结果页均可用。

## 3. 分类监控

`GET /performance-cutover/status` 仅 HR/Admin 可访问，响应按四类隔离：

| 分类 | 字段 | 处理方式 |
|---|---|---|
| 迁移 | `migration.ready/failedItems/blockers` | `ready=false` 时停止发布；按迁移账本业务键定位，不直接改新模型产物 |
| 计算 | `calculation.differenceCount` | 回看最终影子报告和逐人处置记录；不得批量忽略差异 |
| 任务 | `task.staleOpenTasks/activationFailures` | 检查超过 24 小时未完成的已开放任务，以及周期启动检查失败事件、人员权限和任务指派 |
| 通知 | `notification.failedEvents/failedDeliveries` | 先修复 outbox 消费，再按去重键补偿；禁止绕过事件表直接群发 |

告警必须携带 `cycleId`、`participantId`（如适用）、分类和稳定业务键，避免把通知故障误判为业务计算故障。

## 4. 业务验收旅程

主流程分别使用 D、M 已发布表单版本：发布配置 → 创建并启动周期 → SELF/PEER/MANAGER 提交 → 校准 → 发布结果版本 → 员工确认 → 管理员/覆盖全员的 HR 归档。AI 任务是非阻塞参考；提醒时间是软截止，超过后仍允许提交。

整体退回流程：Admin 预览影响并确认 → 已确认结果版本、校准和申诉链失效 → 发出 `RESULT_INVALIDATED` 通知 → 解除人工评估锁并把参与者恢复为 `ACTIVE` → 重新提交 → 重新校准 → 发布新版本 → 再次确认 → 归档。

成功标志：`backend/test/performance-cutover-journey-postgres.integration-spec.ts` 的 D、M、rollback 三条测试全部通过；归档后任何配置、提交、校准、发布或退回写入均被拒绝。

## 5. 回退原则

contract 前可停止部署并继续使用扩展期版本；contract migration 成功后 `rollbackEnabled=false`，不允许把应用切回旧读写，因为旧结构已经删除。此时回退只能是：

1. 停止业务写入。
2. 回滚应用与数据库到 contract 前同一备份点。
3. 修复 readiness、数据或 migration 后，从隔离演练重新开始。

禁止手工重建旧表、把旧枚举值写回参与者状态，或只回滚应用不回滚数据库。

## 6. 最终领域术语

- 参与者状态只表示结果生命周期：`ACTIVE → CALIBRATED → RESULT_PUBLISHED → CONFIRMED`，以及申诉分支、`NO_RESULT`、`WITHDRAWN`。
- 人工进度来自 `PerfEvaluationTask` 与 `PerfEvaluationSubmission`，不再把“待自评/评审完成”写进参与者状态。
- SELF / PEER / MANAGER 使用同一提交模型；AI 使用独立异步报告，不进入人工提交表。
- 员工可见结果、确认状态与归档历史统一以 `PerfResultVersion` 为准；contract 后不再保留旧单行结果投影。
