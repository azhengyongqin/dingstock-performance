# 绩效新模型切换与运维

本手册对应 Ticket 21 的最终 contract 状态。系统只使用版本化配置/表单、周期快照、统一人工提交、阶段结果、校准决定和不可变结果版本；旧模板、旧周期配置列、分离人工评估表及其枚举已经删除。

## 0. fresh 数据库迁移链

历史迁移 `20260716011825` 是在 `20260716030000` / `20260716050000` / `20260716090000` 之后生成的约束重命名，但其目录名在 fresh 安装时会更早排序。为了不修改任何已发布 migration，迁移链使用三段兼容 bridge：

1. `20260716011000_prepare_misordered_constraint_rename` 在 fresh 早期态创建带专用 comment 的临时约束/占位表；成熟库已有真实外键时 no-op。
2. `20260716020000_cleanup_misordered_constraint_rename_bridge` 只删除 comment 与类型都匹配的哨兵对象，绝不删除真实外键。
3. `20260720204000_normalize_misordered_constraint_names` 在链尾核对本地列、引用列和 `ON DELETE/ON UPDATE` 策略后，把真实外键收敛为 Prisma schema 推导名。

三段迁移对“源名和目标名同时存在”、未知约束类型、非 bridge 占位表等混合态一律 fail closed，不猜测、不自动删改业务结构。

fresh 验收必须使用新建的隔离数据库，不得用 `prisma db push` 替代迁移：

```bash
cd backend
DATABASE_URL='<隔离数据库连接串>' pnpm exec prisma migrate deploy
DATABASE_URL='<同一隔离数据库连接串>' pnpm exec prisma migrate diff \
  --exit-code --from-config-datasource --to-schema prisma/schema.prisma
```

成功标志：全部 migration 的 `finished_at` 非空，数据库中不存在 `__fresh_migrate_bridge` 列或 `fresh-migrate-bridge:20260716011825` comment，且 `migrate diff --exit-code` 返回 0。

> 若空库曾在旧代码上失败于 `20260716011825`，第一条 `DROP INDEX` 可能已生效，数据库并非可安全继续的“空库”。对这类无业务数据的一次性库，必须整库丢弃后重新创建；禁止编辑旧 migration，禁止 `prisma migrate resolve`，禁止手工 `INSERT` / `UPDATE` / `DELETE _prisma_migrations`。若数据库已包含业务数据，必须停止部署并从失败前备份恢复，不得按空库处理。

## 1. 切换门禁

contract migration `20260716170000_contract_legacy_performance_model` 会在任何删表前执行硬检查：

1. 必须存在最近一次 `dry_run=false`、`status=COMPLETED` 且 `readiness_report.ready=true` 的全量迁移批次。
2. 迁移账本必须能为所有历史周期定位 `CYCLE_CONFIGURATION` 目标版本。
3. 回填后任何周期缺少 `current_config_version_id` 都会让 migration 整体失败。
4. 全部条件通过后才删除旧表、旧列和旧 enum，并写入 `system_configs.performance.model.cutover`：`phase=CONTRACTED`、`readPath=VERSIONED`、`writePath=UNIFIED_SUBMISSION`、`rollbackEnabled=false`。

成功标志：Prisma migration 成功提交，`performance.system_configs` 中切换配置为最终态，应用构建和 Ticket 21 PostgreSQL 旅程测试通过。

## 2. 上线步骤

1. 备份数据库并记录当前 migration 版本、迁移批次 ID 与 readiness 报告。
2. 在生产快照恢复出的隔离数据库先执行全部 migration 和 `pnpm test:postgres`。
3. 确认切换配置、超时任务和通知失败指标。
4. 生产执行 migration；成功后部署同一提交的 backend 和 web，不允许混用扩展期应用版本。
5. 用 HR/Admin 身份访问 `GET /performance-cutover/status`，保存响应到变更单。
6. 抽查一名 D、一名 M 员工的 SELF / PEER / MANAGER 提交、校准、结果确认与历史查询。

成功标志：状态接口 `ready=true`；旧 `/templates`、旧自评/上级评估接口返回 404；新模板、周期、看板、通知和结果页均可用。

## 3. 当前运行监控

`GET /performance-cutover/status` 仅 HR/Admin 可访问。`ready` 与 `gate` 表示当前实例是否处于 contract 最终态；`monitors` 只包含当前仍在运行的任务和通知两类监控：

| 分类 | 字段 | 处理方式 |
|---|---|---|
| 任务 | `task.staleOpenTasks/activationFailures` | 检查超过 24 小时未完成的已开放任务，以及周期启动检查失败事件、人员权限和任务指派 |
| 通知 | `notification.failedEvents/failedDeliveries` | 先修复 outbox 消费，再按去重键补偿；禁止绕过事件表直接群发 |

告警必须携带 `cycleId`、`participantId`（如适用）、分类和稳定业务键，避免把通知故障误判为任务故障。

## 4. 业务验收旅程

主流程分别使用 D、M 已发布表单版本：发布配置 → 创建并启动周期 → SELF/PEER/MANAGER 提交 → 校准 → 发布结果版本 → 员工确认 → 管理员/覆盖全员的 HR 归档。AI 任务是非阻塞参考；提醒时间是软截止，超过后仍允许提交。

整体退回流程：Admin 预览影响并确认 → 已确认结果版本、校准和申诉链失效 → 发出 `RESULT_INVALIDATED` 通知 → 解除人工评估锁并把参与者恢复为 `ACTIVE` → 重新提交 → 重新校准 → 发布新版本 → 再次确认 → 归档。

成功标志：`backend/test/performance-cutover-journey-postgres.integration-spec.ts` 的 D、M、rollback 三条测试全部通过；归档后任何配置、提交、校准、发布或退回写入均被拒绝。

## 5. 回退原则

contract 前可停止部署并继续使用扩展期版本；contract migration 成功后 `rollbackEnabled=false`，不允许把应用切回旧读写，因为旧结构已经删除。此时回退只能是：

1. 停止业务写入。
2. 回滚应用与数据库到 contract 前同一备份点。
3. 修复 readiness、数据或 migration 后，从隔离演练重新开始。

禁止手工重建旧表、把旧枚举值写回参与者状态，或只回滚应用不回滚数据库。

## 6. 最终领域术语

- 参与者状态只表示结果生命周期：`ACTIVE → CALIBRATED → RESULT_PUBLISHED → CONFIRMED`，以及申诉分支、`NO_RESULT`、`WITHDRAWN`。
- 人工进度来自 `PerfEvaluationTask` 与 `PerfEvaluationSubmission`，不再把“待自评/评审完成”写进参与者状态。
- SELF / PEER / MANAGER 使用同一提交模型；AI 使用独立异步报告，不进入人工提交表。
- 员工可见结果、确认状态与归档历史统一以 `PerfResultVersion` 为准；contract 后不再保留旧单行结果投影。

## 7. 非生产环境精确重置绩效数据

该命令只允许在 `NODE_ENV=development` 的本地或隔离验收库执行，会删除全部 `perf_*` 数据、绩效导出任务和固定绩效审计记录；不会删除 `lark_*`、`role_grants`、`system_configs`。生产、测试、缺失环境标识或其他任意环境无论是否提供确认值都会拒绝执行。

1. 先确认当前连接的是可丢弃的非生产数据库，并完成必要备份。

   成功标志：数据库地址指向本地或隔离库，并已明确使用 `NODE_ENV=development`。

2. 在后端目录执行带精确确认值的命令：

   ```bash
   cd backend
   NODE_ENV=development RESET_PERFORMANCE_DATA_CONFIRM=DELETE_LOCAL_PERFORMANCE_DATA pnpm reset:performance-data
   ```

   成功标志：终端输出 JSON，其中包含 `performanceBefore`、`removedAuditLogs`、`preservedBefore` 与 `preservedAfter`，且受保护表的前后计数一致。

3. 如需重建默认模板、默认配置和默认周期，再执行：

   ```bash
   pnpm seed:baseline
   ```

   成功标志：命令退出码为 0，默认 D/M 表单模板、默认配置模板和“2026年中绩效评定”周期均已生成。

若缺少确认变量、确认值拼写错误，或 `NODE_ENV` 不是规范化后的 `development`，命令应在读取数据库地址和连接数据库前退出且不产生任何写入；这是安全门禁的成功表现，不要绕过。
