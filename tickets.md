# Tickets: 绩效系统大规模功能完善

围绕版本化模板、周期创建与生命周期、统一评估、可解释计算、逐员工校准和结果闭环，完成绩效系统升级；需求基线见 [PRD](.scratch/performance-system-major-enhancement/PRD.md)。

所有 Ticket 默认状态为 `ready-for-agent`。实施时处理 **frontier**：其全部阻塞项均已完成的 Ticket；同一 frontier 中没有依赖关系的工作可以并行，但每次只领取一个 Ticket。

## 01. 建立绩效计算一致性引擎

**Status:** completed

**What to build:** 在现有计算路径旁建立统一、可解释的阶段结果计算能力，使评级和评分都产生标准计算分，并让后续自评、360°、上级评估和迁移校验共享同一套公开计算语义。此 Ticket 是后续纵向功能切片的必要前置重构，不切换现有生产读写路径。

**Blocked by:** None — can start immediately.

- [x] 加权评级根据配置的 S/A/B/C 映射分计算，加权评分直接使用评审人提交的 0～100 分，并同时保留原始输入和计算分。
- [x] 输入最多保留两位小数；中间过程使用精确小数且不舍入；阶段综合分最终按四舍五入保留两位后再映射等级。
- [x] 支持核心维度约束、任一维度约束、受控参数规则及已确认红线强制 C；不允许任意公式或脚本。
- [x] 计算结果可解释到评估项、维度、关系、综合分、初始等级、命中约束和最终阶段等级。
- [x] 建立共享计算契约测试，覆盖评级、评分、区间边界、映射分、精度、约束和红线场景。
- [x] 现有生产计算行为保持可用，未完成迁移切换前不会因新引擎上线而改变历史结果。

## 02. 发布版本化评估表单模板

**Status:** completed

**What to build:** 让 Admin 独立创建和维护适用于不同职级前缀的评估表单模板，并以不可变发布版本提供给配置模板复用。一个版本完整包含 SELF、PEER、MANAGER 和 PROMOTION 子表单。

**Blocked by:** None — can start immediately.

- [x] Admin 可创建、编辑草稿、发布新版本和归档评估表单模板；已发布版本不可原地修改，非 Admin 只能查看可用版本。
- [x] 每个版本声明唯一适用的系统职级前缀，系统能够识别前缀重复、重叠或缺失覆盖。
- [x] 子表单支持受控维度、排序、权重、核心维度、评分/评级项、文本、Markdown、附件和链接等允许组件。
- [x] 每个加权子表单的常规维度权重合计为 100%，恰好一个核心维度，每个常规维度恰好一个计分项，评估项本身不再设置权重。
- [x] 内置 D 环评默认维度：工作贡献与责任担当 35%（核心）、协作沟通与价值观 45%、学习成长与潜力 20%。
- [x] 内置 M 环评默认维度：结果推动与责任担当 40%（核心）、协作沟通与组织影响 35%、领导力与价值观 25%。
- [x] 内置 D 上级评估默认维度为核心业绩 70%（核心）、价值观 20%、职业素养与潜力 10%；M 为核心业绩 50%（核心）、管理绩效 50%。
- [x] PROMOTION 同时支持员工和 Leader 的评估内容边界，不向 360°评审员暴露任何晋升评估项。
- [x] 发布校验和权限行为通过公开模板操作测试验证，数据库约束能够阻止无效层级、重复顺序和孤立明细。

## 03. 发布版本化配置模板

**Status:** completed

**What to build:** 让 Admin 通过配置模板组合准确的已发布评估表单版本，并统一配置等级、阶段计算方式、约束、360°关系权重、相对任务时间和飞书通知规则。

**Blocked by:** 01. 建立绩效计算一致性引擎；02. 发布版本化评估表单模板。

- [x] Admin 可创建、编辑草稿、发布新版本和归档配置模板；发布版本不可原地修改，HR 可查看和选择已发布版本。
- [x] 阶段模式固定为：SELF 直接评级、PEER 加权评级或加权评分、MANAGER 加权评级或加权评分、AI 直接评级；PEER 默认加权评级，MANAGER 默认加权评分。
- [x] S/A/B/C 的符号和顺序不可改；名称、说明、分数区间、映射分和评语要求可在受控边界内配置，默认映射分为 95/85/70/50。
- [x] 管理员可在配置页面调整组织负责人、项目负责人、同部门同事和跨部门协作方权重；默认分别为 30%、30%、25%、15%。
- [x] 四类关系权重均大于 0、不超过 100%、最多两位小数，发布时总和必须严格等于 100%。
- [x] 配置版本绑定准确的已发布表单版本；发布时校验职级前缀覆盖以及阶段模式与计分项类型兼容性。
- [x] 支持配置受控等级约束、相对开始时间、填写提醒时间和通知策略，并能在发布前展示完整校验错误。
- [x] 模板详情可展示版本来源、绑定关系、计算预览及不可用原因；公开 API 行为和关键发布规则有自动化测试。

## 04. 四步创建绩效周期并固化快照

**Status:** completed

**What to build:** HR/Admin 可以从已发布配置模板快速创建绩效周期，通过四步流程完成基本信息、参与人、计划预览和启动检查；周期独立保存所选规则与表单快照，不再现场重复选择各类表单。

**Blocked by:** 03. 发布版本化配置模板。

- [x] 创建流程固定为基本信息、参与人与职级前缀检查、计划预览、启动检查与操作四步，复杂规则放入高级配置。
- [x] 周期不要求单独配置考核期间起止日期，周期名称承担“2026上半年绩效评定”等业务表达。
- [x] 每名参与人的职级前缀必须且只能匹配一个表单版本；不提供兜底表单、员工级手动覆盖或空职级正常路径。
- [x] 周期创建时复制配置、表单、评级、约束、关系权重、时间和通知规则为独立快照，并保留来源版本信息。
- [x] 创建后可保存为 DRAFT，或在通过启动检查后设置为 SCHEDULED；未通过时展示可操作的失败原因。
- [x] DRAFT 与 SCHEDULED 之间允许按权限调整和退回，不生成提前可填写的评估任务。
- [x] 四步流程、模板选择不可用原因、前缀检查和快照独立性具有服务/API 与关键前端流程测试。

## 05. 启动周期并生成评估任务

**Status:** completed

**What to build:** 让已准备好的周期按计划原子启动，生成对应的自评、360°、上级评估和 AI 任务；任务开始时间控制开放，截止时间只负责提醒而不锁死提交。

**Blocked by:** 04. 四步创建绩效周期并固化快照。

- [x] 周期生命周期使用 DRAFT、SCHEDULED、ACTIVE、ARCHIVED；更细的业务进度通过任务和参与人数据派生展示。
- [x] SCHEDULED 到时执行原子启动检查；成功后切为 ACTIVE 并生成任务，失败则保持 SCHEDULED 并通知 HR/Admin 具体问题。
- [x] 任务开始时间是硬开放门槛，开始前不能保存或提交；到达开始时间后方可填写。
- [x] 填写截止时间只触发飞书通知，截止后评估人仍可保存、首次提交、编辑和重新提交。
- [x] 任务开放、填写提醒、启动失败等通知均由可重试事件触发，重复消费不会产生重复业务结果。
- [x] 周期看板能够根据任务事实展示当前进度、缺失项和下一步操作，而不是依赖细粒度周期状态。
- [x] 自动启动成功、失败、并发启动、任务开放门槛和软截止提醒具有自动化测试。

## 06. 统一评估提交并交付员工自评

**Status:** completed

**What to build:** 以统一评估提交模型承载 SELF、PEER 和 MANAGER，让员工按动态 SELF 表单完成自评与晋升材料，并支持提交后的持续编辑和原子替换，不保留答案历史版本。

**Blocked by:** 05. 启动周期并生成评估任务。

- [x] SELF、PEER、MANAGER 共用统一提交主体，当前需要计算、筛选和汇总的评估项结果以关系化数据保存。
- [x] 员工可保存不完整草稿；首次正式提交必须完成自评直接评级、全部必填计分项和其他必填项。
- [x] 首次提交后再次编辑时保留上次有效提交参与计算；新编辑内容作为临时草稿，完整重新提交后原子替换当前有效值。
- [x] 不创建答案历史版本，仅保留一个当前有效提交和一个可选编辑草稿；审计仍记录关键操作人和时间。
- [x] 动态表单支持模板允许的文本、Markdown、附件和链接等组件，并清楚标记草稿、已生效和待重新提交状态。
- [x] 启用晋升评估时，员工在自评中完成 PROMOTION 员工子表单，必填规则随 SELF 一并校验。
- [x] 自评提交、更新、原子替换、无答案历史和开始时间限制通过公开 API 与关键前端流程测试验证。

## 07. 交付 360°关系分配与动态评估表单

**Status:** completed

**What to build:** HR/Admin 能按受控关系为员工分配 360°评审员，评审员按匹配 D/M 模板填写同一套可观察行为维度，并能够像其他评估人一样持续更新当前答卷。

**Blocked by:** 05. 启动周期并生成评估任务；06. 统一评估提交并交付员工自评。

- [x] 支持组织负责人、项目负责人、同部门同事、跨部门协作方四类计算关系，并在分配界面清晰分组。
- [x] 被评审人的当前直属 Leader 不能成为 360°候选人或被成功分配，后端权限和约束不能只依赖前端隐藏。
- [x] 评审员根据被评审人职级前缀获得对应 PEER 表单，不显示员工或 Leader 的晋升评估内容。
- [x] 系统不提供“了解不足/无法评价”操作；未完成的评审只能保持草稿或正式提交。
- [x] 已提交评审关系不可直接删除；替换评审员需要保留可审计的关系变更，且不会误用旧评审人的权限。
- [x] PEER 支持首次提交、提交后编辑和原子重新提交；更新草稿存在时仍以上次有效提交参与计算。
- [x] 评审员只能访问其被分配任务和允许的员工信息；关系分配、Leader 排除、权限和表单边界具有自动化测试。

## 08. 计算 360°角色加权阶段结果

**Status:** completed

**What to build:** 根据已提交 360°答卷生成可解释阶段结果：先在同一关系内平均，再排除没有有效提交的关系并重新归一化权重，最后计算维度、综合分、初始等级和约束结果。

**Blocked by:** 01. 建立绩效计算一致性引擎；07. 交付 360°关系分配与动态评估表单。

- [x] 每名有效评审员的评级按映射分计算，评分直接作为计算分；原始评级或评分与计算分均被记录。
- [x] 每个维度先对同一关系内所有有效评审员做算术平均，再对有效关系做加权汇总。
- [x] 没有任何有效提交的关系不参与计算，剩余权重按原权重占有效权重总和的比例放大至 100%。
- [x] 当有效关系仅为项目负责人 30% 和同部门同事 25% 时，调整后权重分别为 54.55% 和 45.45%，中间计算不提前舍入。
- [x] 未提交草稿不计分，更新未提交时沿用上次有效提交；完全没有有效答卷时生成明确的“无数据”阶段结果，不合成零分或等级。
- [x] 结果持久化并可供授权管理者查看评审人数、有效关系、原始/调整权重、关系均分、维度分、综合分和约束原因。
- [x] 契约测试覆盖同关系多人、多关系缺失、单关系、部分提交、更新草稿、全草稿和小数权重场景。

## 09. 交付上级评估与权威阶段等级

**Status:** completed

**What to build:** 直属 Leader 使用匹配职级的动态 MANAGER 表单评估员工，由系统根据维度和约束计算校准前权威等级；Leader 不再手工选择初始等级。

**Blocked by:** 01. 建立绩效计算一致性引擎；06. 统一评估提交并交付员工自评；08. 计算 360°角色加权阶段结果。

- [x] 当前直属 Leader 获得 MANAGER 任务，可查看允许的自评和 360°汇总参考信息，并填写动态维度、评语和其他必填内容。
- [x] 启用晋升评估时，Leader 完成 PROMOTION 管理者子表单；员工材料和 Leader 结论有明确边界。
- [x] 上级评估支持加权评分或加权评级；系统计算并保存维度结果、综合分、初始等级、约束原因和最终阶段等级。
- [x] MANAGER 阶段等级是校准前权威等级；SELF、PEER 和 AI 仅作参考，不再跨阶段二次加权形成最终等级。
- [x] Leader 可在首次校准前编辑并重新提交；有效提交和临时更新草稿遵循统一替换规则。
- [x] 当前 Leader 才能提交该员工的上级评估，HR/Admin 不得代替 Leader 填写或提交。
- [x] 动态表单、计算结果、必填校验、晋升边界和权限行为具有公开 API 与关键前端流程测试。

## 10. 安全转移直属 Leader 评估职责

**Status:** completed

**What to build:** 当员工直属 Leader 变化或原 Leader 无法继续评估时，HR/Admin 可以明确转移评估责任，同时保留已生效数据和审计，并正确调整校准前后的访问权限。

**Blocked by:** 09. 交付上级评估与权威阶段等级。

- [x] 授权 HR/Admin 可选择新的评估 Leader，操作必须记录原因、原负责人、新负责人、操作者和时间。
- [x] 首次校准前更换 Leader 时，新 Leader 获得编辑权；旧 Leader 的有效 MANAGER 提交继续生效，直到新 Leader正式重新提交。
- [x] 新 Leader 的编辑草稿不能提前覆盖旧的有效提交，正式提交时完成原子替换。
- [x] 首次校准后更换 Leader 不解锁或改写历史评估与校准，仅转移未来敏感信息查看和重新校准权限。
- [x] 旧 Leader 在职责转移后不能继续编辑、校准或查看其已无权访问的敏感信息。
- [x] 转移操作更新待办和必要通知，且在并发更换或提交时不会出现双负责人。
- [x] 校准前后转移、有效提交保留、旧权限撤销和并发控制具有自动化测试。

## 11. 将 AI 评估适配为独立异步参考

**Status:** completed

**What to build:** 使用统一后的自评、360°和上级评估输入生成 AI 报告及参考等级，同时保持 AI 为独立异步能力，不成为周期或参与人的阻塞阶段。

**Blocked by:** 08. 计算 360°角色加权阶段结果；09. 交付上级评估与权威阶段等级。

- [x] AI 任务从当前有效人工评估和阶段结果构建输入，并记录输入修订信息，避免使用更新草稿替代有效提交。
- [x] AI 报告拥有独立的等待、运行、成功和失败状态；失败、超时或未生成都不阻止校准、结果发布或归档。
- [x] 人工有效输入变化时按规则触发重新生成或标记过期，不把 AI 变化视为人工评估修订冲突。
- [x] AI 只产生直接参考等级和报告，不参与 SELF、PEER、MANAGER 的跨阶段权重计算。
- [x] AI 内容只向获授权管理角色提供，员工结果接口和页面不返回 AI 报告、等级或生成状态。
- [x] 任务幂等、失败重试、输入更新、非阻塞校准和员工隐私边界具有自动化测试。

## 12. 处理必交评估与“当前周期无绩效结果”

**Status:** completed

**What to build:** 在进入校准前明确处理缺失的自评和上级评估：SELF 与 MANAGER 都是必交项；只有长期缺失自评时，HR/Admin 才能把员工关闭为当前周期无绩效结果。

**Blocked by:** 05. 启动周期并生成评估任务；06. 统一评估提交并交付员工自评；09. 交付上级评估与权威阶段等级。

- [x] 首次校准前必须存在完整有效的 SELF 和 MANAGER 提交；PEER 与 AI 缺失不会阻塞校准。
- [x] 缺失 SELF 时，授权 HR 或 Admin 可填写原因并把参与人标记为 NO_RESULT，界面文案显示“当前周期无绩效结果”。
- [x] NO_RESULT 不生成绩效结果、不是退出周期，并在归档前允许 HR/Admin 撤销后继续补交与评估。
- [x] 缺失 MANAGER 时不允许设置 NO_RESULT；系统持续显示阻塞并要求升级提醒或更换评估 Leader，任何人不得代交。
- [x] 员工首次从未提交时保留草稿并固化为无数据阶段结果；已有有效提交但更新未提交时仍以上次有效提交为准。
- [x] 参与人状态只记录结果生命周期，评估缺失和任务进度从对应任务与提交派生。
- [x] 必交门槛、NO_RESULT 权限与撤销、缺失 MANAGER 阻塞和无数据语义具有自动化测试。

## 13. 逐员工校准、锁定与并发控制

**Status:** completed

**What to build:** 让直属 Leader、授权 HR 和 Admin 对每名员工执行明确的 KEEP 或 ADJUST 校准，并在首次成功校准时逐员工锁定人工评估，防止基于过期输入覆盖他人决定。
